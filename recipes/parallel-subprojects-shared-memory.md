# Параллельные подпроекты: общая память, изолированный контекст

## Как вызвать

Скажите Claude Code в обычном чате:

> «онбордь подпроект <имя>» · «хочу работать над подпроектами параллельно, с общей памятью» · «настрой подпроект так, чтобы он видел общую память, но не тащил чужой контекст»

Точная формулировка не важна. Claude Code проверит структуру проекта и поставит штамп из трёх файлов (описан ниже).

## Проблема, которую это решает

Крупный проект разрастается, и появляется потребность работать над несколькими его подпроектами **параллельно** — каждое окно Claude Code со своим контекстом, но с одной общей памятью и общими правилами.

Работая из корня проекта, вы тащите в одно окно контекст всех подпроектов сразу — параллелить невозможно, контекст раздут. Наивное «просто запускать из подкаталога» даёт изоляцию, но ломает две вещи:

1. **Рвёт общую память.** Вне git-репозитория Claude Code считает каждый каталог запуска отдельным проектом и заводит ему свой пустой силос памяти. Уроки, накопленные проектом, подпроекту не видны; его находки не попадают в общую копилку.
2. **Крадёт идентичность.** Подпроект без своего CLAUDE.md наследует личность головного проекта — новая сессия считает себя «всем проектом», читает головной статусный файл и работает не с тем контекстом.

## Ключевые факты механики

Что доходит из корня вниз в подпроект при запуске из подкаталога **вне git-репозитория**, что нет. Пометки: «проверено» — подтверждено живым тестом; «наблюдение» — живое поведение на одной версии, документацией не закреплено; без пометки — по документации Claude Code:

| Компонент | Наследуется вниз? | Механизм |
|---|---|---|
| CLAUDE.md (правила, конвенции) | ДА (проверено) | чтение вверх по дереву каталогов от точки запуска |
| Скилы `.claude/skills/` | ДА (проверено) | загружаются из каждого родительского каталога до корня |
| Агенты `.claude/agents/` | ДА (по документации) | тот же обход вверх от текущего каталога |
| `settings.local.json`: permissions | НЕТ (наблюдение) | настройки предка из подкаталога не читаются |
| `settings.local.json`: SessionStart-хук | НЕТ (наблюдение) | тот же резолвер; хук предка в подкаталоге не срабатывает |
| Общая авто-память | НЕТ автоматом | нужен `autoMemoryDirectory` в настройках подпроекта |
| Локальная идентичность | нет по умолчанию | подпроект наследует личность головного, пока нет своего CLAUDE.md |

**В git-варианте таблица другая:** внутри git-репозитория корень репо — это и есть «проект», поэтому настройки корня (permissions, хуки) в подкаталогах применяются, а память едина без всякой настройки. Строки «НЕТ» выше относятся к не-git дереву.

Дополнительные документированные факты:

- **`autoMemoryDirectory`** (settings.json, любой scope, абсолютный путь) редиректит и **чтение, и запись** авто-памяти на заданную папку. Именно это даёт подпроекту общую память. Механизм авто-памяти документирован с Claude Code 2.1.59; сам ключ `autoMemoryDirectory` появился в **2.1.74** — версию проверять по второму порогу.
- **Альтернатива — git-граница.** Все worktree и подкаталоги одного git-репозитория нативно делят одну папку авто-памяти: `git init` в корне объединяет память подкаталогов без единой настройки. Git здесь — только «граница проекта», не версионирование. Выбор между способами — по проекту: для крупного не-кодового или сетевого дерева `autoMemoryDirectory` не трогает дерево git-ом.
- **CLAUDE.md каталога запуска читается последним** (высший приоритет) → тонкий `<подпроект>/CLAUDE.md` перекрывает и дополняет корневой, давая локальную личность.
- **Лимит загрузки индекса памяти:** первые 200 строк / 25 КБ MEMORY.md (см. [`setup-memory.md`](setup-memory.md)) — общий индекс на несколько подпроектов упирается в него раньше.

## Рецепт: штамп «онбордить подпроект» (три файла)

Параметры: `ROOT` — корень проекта; `SUB` — каталог подпроекта; `MEM` — абсолютный путь к общей папке памяти. По умолчанию — существующая папка авто-памяти корня: её путь виден в системном контексте сессии, запущенной из `ROOT`. Можно выбрать и отдельный канонический путь.

### 1. `SUB/.claude/settings.local.json` — общая память

```json
{
  "autoMemoryDirectory": "<MEM>"
}
```

Мержить в существующий файл, **не затирать** (сессия могла дописать свои ключи). Значение `<MEM>` — абсолютный путь или `~/...`. Активируется после диалога доверия к папке при первом запуске. Сюда же при желании — permissions подпроекта (вниз они не наследуются, см. таблицу).

Ту же настройку стоит закрепить и в `ROOT/.claude/settings.local.json` **корня**: тогда путь памяти зафиксирован у всех участников явно и не зависит от внутренней схемы имён каталогов Claude Code — если резолвер сменит схему, корень и подпроекты не разъедутся по разным папкам.

### 2. `SUB/CLAUDE.md` — локальная идентичность (тонкий, без пересказа состояния)

```markdown
# Подпроект <Имя> — часть <родитель>

Ты работаешь в подпроекте <Имя>: <одна строка сути>.
Первым делом прочитай своё рабочее состояние — ./README.md.
Правила <родителя>, скилы, агенты и общая память наследуются от корня —
не пересказывать. Головной статусный файл читать только если задача выходит
за рамки подпроекта.
```

### 3. `SUB/README.md` — файл состояния подпроекта

Владелец изменчивых фактов подпроекта (материалы, статус, ближайший фокус) — по правилу «один факт — один владелец» ([`fact-ownership.md`](fact-ownership.md)). Если уже есть — не трогать. CLAUDE.md на него ссылается, не дублируя.

Практики (скилы, агенты, корневой CLAUDE.md) штамп **не трогает** — они наследуются вниз сами.

## Проверка (канарейка)

Запустите свежую сессию из `SUB` и спросите:

> «Над каким проектом работаешь, где твоё состояние, какой файл главный, видишь ли общую память — назови указатель из индекса?»

**Правильный ответ:** «подпроект <Имя>, состояние — SUB/README.md» + называет реальные указатели из общего индекса памяти.
**Неправильный** (идентичность не задана): «проект — головной, главный файл — корневой статусный».

Вторая канарейка — на **запись**: попросите сессию подпроекта запомнить тестовый факт и проверьте, что файл лёг в общую папку памяти, а не в пустой силос подпроекта.

Третья канарейка — на **наследование**: спросите, какие правила и скилы сессия видит — корневые должны присутствовать. Walk-up вне git-репозитория документирован не для всех компонентов и между версиями меняется, поэтому проверка обязательна.

## Границы применимости (за ними схему менять)

- **Память:** плоский общий индекс держится до ~250–350 файлов (лимит 25 КБ / 200 строк загрузки). Дальше — иерархия индексов, retrieval по темам вместо «читать всё».
- **Состав:** модель «всё общее, без фильтрации» держится на **одном пишущем операторе**. Второй человек ломает три вещи: приватность (секреты в общей памяти под git — утечка → строгий `CLAUDE.local.md`/gitignore), конкурентную запись (merge-конфликты индекса), конфликт знаний (два противоположных урока без приоритета). Дешёвая страховка заранее: держать во frontmatter памяти поля `origin`/`scope` как выключенные заготовки — переход на команду станет перекладкой строк.
- **Топология:** sibling-репозитории без общего родителя — `autoMemoryDirectory` на внешний абсолютный путь + общие правила через `--add-dir` (с переменной окружения `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`, иначе правила из добавленного каталога молча не грузятся) или симлинк `.claude/rules/`. Разные семьи проектов — **не** ставить `autoMemoryDirectory` на user-level (потащит одну память во все проекты машины); ставить в настройки подпроектов своей семьи.
- **Межмашинность:** авто-память документирована как machine-local. Общий путь на сетевой шаре (одинаковая буква диска / точка монтирования на всех машинах) делает её фактически общей — off-label, работает потому, что путь физически один.
- **Версии:** механика проверялась вживую на одной версии Claude Code и одной ОС; резолвер настроек и памяти между версиями меняется (см. тупик про path-keyed имена ниже). После настройки — прогнать канарейки, не верить на слово.

## Что отвергнуто и почему (чтобы не повторять тупики)

Паттерн прошёл несколько раундов состязательного разбора. Отвергнутое:

- **Отдельная папка-канон `_shared` + свод файлов памяти туда.** Не нужно: нативный авто-механизм привязан к своей папке; `autoMemoryDirectory` редиректит его штатно, без переноса файлов. Ручная запись в `_shared` рвёт связь с авто-механизмом.
- **User-level / project-level хук доставки индекса памяти.** Не работает для вложенности как задумано: project-хук предка в подкаталоге не срабатывает (наблюдение на проверенной версии — поведение резолвера между версиями может меняться); user-level тащит одну память во все проекты машины. Плюс индекс не влезает в лимит инъекции хука. При глубоком дереве отдельно учесть: резолвер авто-памяти в некоторых версиях уходит вверх к предковой папке вместо своей ([issue #53734](https://github.com/anthropics/claude-code/issues/53734)) — после настройки прогнать канарейку из подпроекта.
- **Метки `origin`/`scope` + фильтрация в хуке.** Преждевременно для одного оператора (мультитенантность без второго человека). Сами поля во frontmatter оставить можно как заготовку, но фильтрацию не строить.
- **Опора на конкретное имя path-keyed каталога памяти** — служебного каталога, куда Claude Code складывает авто-память проекта, с именем, выведенным из пути проекта (`<путь-через-дефисы>`). Хрупко: это деталь реализации, резолвер нестабилен между версиями. Опираться на `autoMemoryDirectory` (явный путь) или git-границу, не на выведенное имя.
- **Разрезание головного статусного файла по подпроектам как обязательный шаг ради общей памяти.** Не связано: память решается настройкой, статусные файлы — отдельно (их деление — вопрос [`fact-ownership.md`](fact-ownership.md), не этого рецепта).

## Итог одной формулой

Запуск из подкаталога = изолированный контекст (проверено). `autoMemoryDirectory` в настройках подпроекта (или git-граница в корне) = общая память — чтение и запись проверены вживую. Тонкий CLAUDE.md подпроекта = локальная идентичность (проверено). Правила и скилы наследуются вниз сами (проверено), агенты — по документации, тем же обходом вверх. Три файла на подпроект, одна команда онбординга; миграция памяти не нужна.

### Шаги онбординга подпроекта

1. **Проверь версию и границу проекта.** `claude --version` >= 2.1.74 (порог появления ключа `autoMemoryDirectory`; сам механизм авто-памяти — с 2.1.59). Выясни, git-репозиторий ли корень: если да — общая память уже работает нативно, штамп сокращается до двух файлов (CLAUDE.md + README, без settings).
2. **Определи параметры со слов пользователя:** SUB — каталог подпроекта, MEM — общая папка памяти. По умолчанию — существующая папка авто-памяти корня: её путь виден тебе в системном контексте сессии, запущенной из корня. Покажи пользователю, какой путь выбрал.
3. **Фильтр каталогов:** штамповать только реальные каталоги-эффорты. НЕ онбордить служебные (node_modules, assets, docs, кэши) и вложенные чужие git-репозитории.
4. **Поставь файлы штампа** (два или три — по итогам шага 1) по рецепту выше: settings.local.json (мерж, не перезапись; в не-git варианте — плюс тот же ключ в настройки корня), тонкий CLAUDE.md (если пользователь уже писал свой — не перезаписывать, предложить дополнение), README.md (если есть — не трогать).
5. **Прогони канарейки — с участием пользователя.** `autoMemoryDirectory` активируется только после интерактивного диалога доверия к папке: попроси пользователя один раз запустить обычную сессию из SUB и подтвердить доверие. После этого прогони канарейки (идентичность, запись, наследование) — или продиктуй пользователю вопросы, если запускать из SUB можешь только он. Результат покажи до объявления успеха.
6. **Раскатывай инкрементально:** один подпроект → канарейки → следующий. Не пачкой.

### Строгие правила (что НЕ делать)

1. **НЕ перезаписывай существующие файлы** — settings.local.json только мержить, рукописный CLAUDE.md и README.md подпроекта не трогать без явного согласия.
2. **НЕ ставь `autoMemoryDirectory` на user-level** — одна память расползётся на все проекты машины.
3. **НЕ опирайся на имя path-keyed каталога памяти** — только явный `autoMemoryDirectory` или git-граница.
4. **НЕ объявляй успех без канареек** — резолвер настроек и памяти меняется между версиями; проверка обязательна.
5. **НЕ переноси и не копируй файлы памяти** — редирект решает задачу без миграции; ручной свод рвёт связь с авто-механизмом.
