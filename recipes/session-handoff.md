# Сеансовая эстафета — передача контекста новой сессии

## Как вызвать

Скажите Claude Code в обычном чате одну из фраз:

> «давай сделаем хэндофф» · «собери эстафету» · «собери session handoff по сессии» · «давай зафиксируем состояние перед паузой»

Точная формулировка не важна — модель понимает по смыслу. Это не slash-команда, не нужен префикс `/`. Просто обычный запрос на естественном языке в любой момент, когда вам нужна эстафета (конец дня, перед `/compact`, перед отпуском, перед передачей коллеге).

Claude Code задаст 5 коротких вопросов по ядру (главный блокер, от кого ждёте решение, ключевые сдвиги, запреты, первая задача) и соберёт файл `docs/SESSION-HANDOFF-YYYY-MM-DD.md`. Если уже есть SessionStart hook (см. ниже) — следующая чистая сессия сама прочитает этот файл при старте.

## Главный принцип

**Эстафетный файл — не отчёт о сделанном, а исполнимая программа холодного старта.** За 10 минут чтения новая сессия должна назвать пять открытых вопросов, актуальные версии артефактов и первое действие — без re-read истории. Если для этого нужен один абзац — пишите один абзац, если пять — пишите пять. Подстраховаться объёмом — провалить эстафету: 60% страниц никто не читает.

Принцип проверен на нескольких рабочих проектах — разработка, юридическая практика, бухгалтерия с аналитикой. На каждом приходили к одной схеме самостоятельно: read-протокол с гарантированной доставкой + компактное ядро + опциональные расширения по профилю.

## О чём этот рецепт

Сеансовая эстафета (session handoff) применяется перед `/compact`, перед новой сессией, в конце рабочего дня, при возврате к проекту после паузы, при передаче работы коллеге или себе-будущему через две недели. Метафора прямая: эстафетная палочка. Уходящая сессия передаёт следующей всё, что нужно для продолжения.

## Граница с `track-progress.md`

Эти два рецепта часто путают. Граница жёсткая:

| | `track-progress.md` | Сеансовая эстафета |
|---|---|---|
| **Что это** | Маркер «где я остановился» | Полный снимок состояния |
| **Размер** | Одна строка в CLAUDE.md | 200–400 строк отдельного файла |
| **Когда** | Автоматически, на лету | Осознанно, на смене фазы или в конце дня |
| **Триггер** | Короткая пауза, следующий шаг | Конец дня, отпуск, `/compact`, передача |
| **Аналогия** | Закладка в книге | Досье по делу |

**Алгоритм выбора:**
- Сессия меньше 2 часов, нет архитектурных сдвигов → `track-progress`.
- Сессия дольше 2 часов или закрыта значимая фаза → эстафета.
- Уходите в отпуск, передаёте коллеге, ожидаете `/compact` → эстафета обязательна.

Эстафета не заменяет `track-progress`, а дополняет его. Если в проекте уже настроен hook от `track-progress`, для эстафеты добавляется второй hook рядом, не вместо (см. ниже).

## Три правила, без которых эстафета не работает

1. **TL;DR пишется последним.** Сначала — все остальные секции. Только когда понятна полная картина, формулируется ядро §0. Если §0 написать первым, он превращается в план «что я хотел сделать», а не «что важно для следующей сессии».
2. **Цифры live-verified, не из памяти.** Любая версия, счётчик, размер — пробивается на момент написания (`git rev-parse`, запрос к сервису, SQL-запрос). Копировать из предыдущей эстафеты нельзя — старая цифра подрывает доверие к документу.
3. **Готово, а не что делал.** Процесс («провёл анализ», «сделал три ревью») идёт в журнал. В эстафете — снимок результата: что закрыто, что открыто, какие решения приняты.

## Структура — восемь обязательных разделов

Все восемь разделов применимы в любом домене. §3 и финальный чеклист масштабируются от трёх строк до десятка — короткая сессия может обойтись минимумом, длинная инженерная работа разрастается за счёт деталей.

### §0 — TL;DR за 60 секунд

Пять-восемь пунктов, каждый — одно предложение. Где застряли, чего ждём, что главное за период, красные флаги, первая задача. Именно §0 инжектируется через hook в новую сессию, поэтому суммарно §0+§1 не превышают 80 строк (~3–5 КБ).

Проверка качества §0: дайте его прочитать стороннему человеку. Если за 60 секунд он не может сказать «эти ребята застряли на X, ждут Y» — §0 написан как план, не как эстафета.

### §1 — Первые 10 минут новой сессии

Буквальный пронумерованный чеклист с конкретными командами и путями. Не «ознакомиться с состоянием», а «выполнить `git fetch && git status`, прочитать `docs/decisions/2026-04-pricing.md`, проверить ленту дел СПАРК». Каждый шаг атомарен — выполняется без интерпретации.

Дополнительно — две короткие подсекции:
- **Что спросить у оператора** (3–5 вопросов).
- **НЕ начинать без напоминания** (3–5 запретов: «не подавать жалобу до экспертизы», «не мёрджить #142 до code review»). Внутри §1 остаются только запреты, нарушение которых — инцидент. Всё прочее — за ссылкой.

### §2 — Состояние проекта (live-verified снимок)

Версии артефактов, источники данных, ключевые числа, открытые и закрытые циклы. Каждая цифра — на момент написания эстафеты. Для разработки: версии репозиториев, SHA, статус прода. Для бухгалтерии: размер БД, число контрагентов, период отчётов. Для юриспруденции: статусы дел, даты заседаний.

Если вместо «версии» уместно слово «артефакты» или «источники» — используйте слово домена. Называйте вещи так, как принято в команде, а не так, как написано в шаблоне.

### §3 — Что закрыто

Группировка **по фичам или решениям**, а не по коммитам/датам. Один период работы может закрыть три задачи, одна задача — растягиваться на пять сессий. Эстафета фиксирует логические единицы. Каждая закрытая фича — заголовок + 2–3 строки результата + ссылка на детали (если есть). Не пересказ работы, а её итог.

### §4 — Что открыто

Приоритизированный список: P0 → P1 → P2 → P3, или «срочно / на неделю / на месяц», или «блокирует / уточнить / доработать» — формат под домен. Главное — приоритизация и **связи между задачами**: P1-1 → P1-3 (одна тянет другую). Без связей список становится плоским и теряется стратегическая нить.

Оценки только в часах с пометкой S/M/L: «~2–3 ч, S», «полдня, M». Дни и недели здесь не работают — сессия не знает, сколько реального времени займёт работа.

### §5 — Грабли и уроки

Каждая грабля: что случилось, как обнаружено, как решено, как предотвратить. Раздел делится на два уровня:

- **Грабли сессии** — то, что обнаружено в текущем периоде. Если не записать — теряются вместе с контекстом.
- **Исторические грабли (H1, H2…)** — постоянные ловушки, переносятся между эстафетами без изменений. Обновляются раз в недели или месяцы, не каждую эстафету.

После 3–5 повторений грабля мигрирует в reference-память проекта (см. [`setup-memory.md`](setup-memory.md)) и из эстафеты убирается.

### §6 — Конкретные вопросы оператору

5–10 вопросов, требующих решения именно от пользователя, а не от исполнителя. Формат: actionable. Не «обсудить ценообразование», а «подтвердить от Андрея: можно ли поднять цены на тариф B на 15% до 1 мая». Если вопрос завис на трёх эстафетах подряд — переформулируйте его или закройте принудительно. Долго висящие вопросы либо неважны, либо плохо сформулированы.

### Финальная сверка (red flags)

Перед тем как считать эстафету готовой — пройдитесь по чеклисту. Без галочки `[x]` — эстафета не готова, дописывайте. На крупных проектах раздел разрастается до 20–30 пунктов; минимальный набор:

- [ ] Версии артефактов проверены live, не из памяти
- [ ] §0 TL;DR проходит 60-секундный тест на стороннем читателе
- [ ] §1 — буквальные команды, не «ознакомиться»
- [ ] Все P0 в §4 имеют ID и оценку S/M/L
- [ ] Грабли сессии перенесены в §5, не остались в голове
- [ ] Открытые вопросы §6 — конкретные, без «обсудить»
- [ ] Hook-инжекция работает (тест A/B пройден)
- [ ] CLAUDE.md или LATEST-pointer ссылается на актуальную эстафету

## Шаблон §0 + §1

Эти две секции — то, что инжектируется через hook. Держите их в пределах 80 строк суммарно.

### Универсальный шаблон

```markdown
# Сеансовая эстафета — <название проекта> (YYYY-MM-DD)

## §0. TL;DR за 60 секунд

1. **Где застряли:** <одно предложение — главный блокер>
2. **Чего ждём:** <от кого какое решение>
3. **Главное за период:** <2-3 крупных сдвига>
4. **Красные флаги:** <критичные запреты>
5. **Первая задача:** <что делать сразу после прочтения>

## §1. Первые 10 минут

**Обязательные команды/проверки:**
1. <команда 1 с конкретным путём>
2. <команда 2 с конкретным путём>
3. <команда 3>

**Что спросить у оператора:**
- «<вопрос 1>»
- «<вопрос 2>»

**НЕ начинать без напоминания:**
- <запрет 1 — нарушение = инцидент>
- <запрет 2>

**Если нужен детальный контекст** — читать полный файл
`docs/SESSION-HANDOFF-YYYY-MM-DD.md` (разделы §2–§11).
```

### Пример: разработка

```markdown
# Сеансовая эстафета — Project Alpha (2026-04-14)

## §0. TL;DR за 60 секунд

1. **Где застряли:** миграция БД блокирована ожиданием подтверждения DBA по RLS-политике.
2. **Чего ждём:** ответ DBA до конца недели, иначе откатываем на v0.2.
3. **Главное за период:** auth-модуль закрыт (1129 vitest GREEN), purchase-flow на новой схеме.
4. **Красные флаги:** не катить миграцию в прод до закрытия P0 C0-02; не трогать `src/payments/*`.
5. **Первая задача:** проверить ответ DBA → если есть, запустить миграцию в стейджинг.

## §1. Первые 10 минут

**Команды:**
1. `git fetch && git status` — проверить, есть ли новые правки
2. `gh pr view 142 --json state` — статус PR миграции
3. `cat docs/decisions/2026-04-rls.md` — последние решения по RLS

**Спросить:**
- «DBA ответил по RLS?»
- «Можно катить в стейджинг без clean rebuild?»

**НЕ начинать:**
- не мёрджить #142 до code review
- не править `src/payments/*` (заморожено до релиза 0.3)
```

## Надёжная доставка через SessionStart hook

*(Раздел для разработчиков или тех, кто настраивает Claude Code с их помощью.)*

### Зачем механизм, а не текст в CLAUDE.md

Одна ссылка в CLAUDE.md «читай эстафету первым делом» не гарантирует прочтения. Это [правило 7 в `behavior-rules.md`](behavior-rules.md): инструкция в тексте — совет, который модель может отклонить в пользу короткого пути. Надёжность даёт **mandatory-trigger** — SessionStart hook, выполняемый Claude Code до первого ответа модели; его вывод инжектируется в `system-reminder` сессии и физически попадает в контекст.

### Invariant pointer LATEST — рекомендуемый подход

Вместо поиска свежей эстафеты по дате (`ls -t docs/SESSION-HANDOFF-*.md | head -1`) надёжнее использовать **указатель-инвариант**:

```
docs/SESSION-HANDOFF-LATEST.md   ← всегда указывает на актуальную эстафету
docs/SESSION-HANDOFF-2026-04-14.md  ← фактический файл за конкретную дату
```

При создании новой эстафеты обновляется только LATEST. На Linux и macOS — через симлинк (`ln -sf 2026-04-14.md SESSION-HANDOFF-LATEST.md`). На Windows проще копировать: `cp docs/SESSION-HANDOFF-2026-04-14.md docs/SESSION-HANDOFF-LATEST.md` — симлинки требуют Developer Mode или admin-прав, а это блокер для junior-команд. Hook читает файл, а не следит за ссылкой, поэтому копия и симлинк работают одинаково.

Минус копирования: при правке исходного файла нужно вручную обновить LATEST. Если регулярно забываете — заведите алиас `handoff-publish` или git pre-commit hook, который сам копирует. Без автоматизации LATEST через неделю устаревает молча.

Преимущество схемы с LATEST: невозможна ошибка с двусмысленностью имён (`2026-05-07.md` vs `2026-05-07-inventory.md`), и CLAUDE.md не требует правок при каждом новом handoff.

### Готовая команда hook

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "f=docs/SESSION-HANDOFF-LATEST.md; [ -f \"$f\" ] && head -n 80 \"$f\" | jq -Rs '{hookSpecificOutput: {hookEventName: \"SessionStart\", additionalContext: (\"# SESSION HANDOFF TL;DR\\n\\n\" + .)}}' 2>/dev/null || true",
            "timeout": 10,
            "statusMessage": "Загрузка эстафетного файла..."
          }
        ]
      }
    ]
  }
}
```

Команда ищет LATEST, берёт первые 80 строк (§0+§1), оборачивает в JSON и отдаёт в system-reminder. `|| true` — страховка: если файла нет, сессия стартует штатно без ошибки.

Лимит вывода — 10 000 символов. Ядро §0+§1 (~3–5 КБ) гарантированно влезает; полный файл (200–400 строк) — нет, поэтому через hook идёт только ядро. Полный файл модель подчитывает через `Read`, когда вопрос требует деталей §2 и далее.

Куда положить hook:
- `.claude/settings.local.json` — личная настройка, допускает абсолютные пути, не коммитится в git.
- `.claude/settings.json` — командная, коммитится, использует только относительные пути.

### Зависимости и ограничения

`jq` не встроен в Git Bash на Windows и не входит в стандартный macOS — поставьте отдельно (`choco install jq` для Windows, `brew install jq` для macOS, пакетный менеджер для Linux). PowerShell и cmd.exe не подходят: команда написана для POSIX-совместимого shell — Git Bash или WSL.

Если в проекте уже есть SessionStart hook (от `track-progress.md`, AI Factory или другой настройки) — **не заменяйте, а добавляйте** второй элемент в массив `SessionStart[0].hooks[]`. Перед записью прочитайте текущий settings.

### Подводные камни на Windows и в монорепе

Эти случаи в стандартный hook не заложены — добавляйте под свой проект:

- **CRLF-окончания строк.** Файлы, отредактированные нативными Windows-редакторами, содержат `\r\n`. После `head -n 80` они попадают в jq и появляются в system-reminder склеенными или с видимыми `\r`. Лечится добавлением `tr -d '\r'` перед `jq`: `head -n 80 "$f" | tr -d '\r' | jq -Rs ...`.
- **Симлинки на Windows.** `ln -s` требует Developer Mode или прав администратора. Проще делать LATEST через копирование: `cp docs/SESSION-HANDOFF-2026-04-14.md docs/SESSION-HANDOFF-LATEST.md`. Минус: при правке исходника нужно обновить копию. Решается алиасом `handoff-publish` или git pre-commit hook'ом.
- **Двое разработчиков на одном репо.** Один LATEST-файл на проект — узкое место: коллега перезаписывает ваш handoff своим. Решение: `SESSION-HANDOFF-LATEST-<имя>.md` с пользовательским hook, или правило «один handoff в день, второй человек дописывает».
- **Монорепа с несколькими сервисами.** Один LATEST на корень тянет в сессию неактуальный для текущего сервиса контекст. Заведите `<service>/docs/SESSION-HANDOFF-LATEST.md` и hook, который выбирает по `$PWD` (или `$CLAUDE_PROJECT_DIR`).
- **Worktrees.** При `git worktree add` settings копируются, но docs/ принадлежит одному дереву. В feature-ветке hook покажет эстафету из main. Если работаете через worktrees, держите эстафету per-worktree или отключайте hook в feature-ветке.
- **Timeout.** На холодном диске на Windows 10 секунд бывает мало. Если `statusMessage` мигает дольше — увеличьте до 15.

### Самопроверка hook

После настройки — обязательная A/B-проверка:

1. `/exit` → `claude` в корне проекта (новая чистая сессия).
2. Контрольный вопрос: «статус работы по проекту?». Не подсказывайте про эстафету.
3. В ответе должны быть факты из §0 (главный блокер, красные флаги). Если их нет — hook не сработал, см. диагностику в приложении.

## Адаптация под тип проекта

Восемь обязательных разделов применимы в любом домене. Опциональные расширения добавляются под специфику.

Для **data, аналитики, исследований** ключевая боль — параллельные темы. Заведите таблицу «Параллельные ветки»: 5–8 одновременных направлений, между которыми вы переключаетесь (запросы, отчёты, гипотезы). Без неё новая сессия видит только последнюю активную тему и начинает «откапывать», что было раньше. Полезна и карта Memory — список reference-файлов проекта с колонкой «когда читать».

Для **длинных документов** (договоры, регламенты, методички) — отдельный файл EDIT-LIST с таблицей правок: ID / где / что было / что меняется / источник. Эстафета на него ссылается, не дублирует. Это освобождает основной файл от тысячи мелких правок и позволяет применять их пачками.

Для **multi-stakeholder ревью** (от трёх ЛПР и больше) — карточки персон: операционные памятки по каждому, возражения, что закрывает, как разговаривать. Не художественные портреты, а инструкция «как пройти возражения этого человека».

Если данные для эстафеты собирают параллельные агенты — финальный синтез делает основной поток сессии, не subagent. Субагент видит только свои отчёты, теряет тон оператора и реальные решения. Операционные гарантии параллельных запусков — в [`parallel-agents-watchdog.md`](parallel-agents-watchdog.md).

## Жизненный цикл эстафет: архив и реестр

На живом проекте эстафеты накапливаются быстро (20+ версий за месяц — реальный темп). Без жизненного цикла папка docs/ превращается в свалку, а вопрос «какая версия актуальна» получает несколько ответов.

**Архив + реестр.** Старые эстафеты — в папку архива. Рядом — INDEX.md: версия, дата, файл, одно предложение сути. Реестр без верхней границы («все версии — см. INDEX»); никогда не писать диапазон «от X до Y» в других документах — он протухнет при следующем выпуске.

**Чек-лист новой версии эстафеты** (канонический — другие рецепты ссылаются сюда):
1. Старую версию — в архив.
2. Строку — в INDEX.
3. Волатильные факты среды в новой эстафете — ссылками на reference-память, не значениями.
4. Прогнать самопроверку документации, если настроена (раздел «Самопроверка» в [`fact-ownership.md`](fact-ownership.md)).

**Анти-паттерн: пересказ эстафеты в CLAUDE.md.** Прямое следствие архитектуры этого рецепта: hook доставляет ядро §0+§1 механизмом — дублировать содержимое эстафеты в CLAUDE.md не нужно И вредно. Эмпирика: пересказ в CLAUDE.md пережил 4 версии эстафеты и дезинформировал старт каждой сессии 8 дней подряд, при том что его добросовестно «актуализировали». В CLAUDE.md — только триггер «читать эстафету первым» и механизм доставки, без значений. Положительный пример: проект с чистым LATEST-pointer в CLAUDE.md через месяц интенсивной работы не имел ни одного протухшего факта в нём — указатель не протухает, потому что не содержит значения. Общий принцип — [`fact-ownership.md`](fact-ownership.md).

**Совместимость схем.** LATEST-pointer (описан выше) и схема «активный файл в корне + папка архива» — оба валидны. Реестр INDEX и чек-лист новой версии нужны в обеих.

## Когда эстафета НЕ нужна

Если сессия короче двух часов, задача разовая (исследование, расчёт, ad hoc анализ) или в проекте нет CLAUDE.md с настроенным hook — эстафета не поможет: либо нет что передавать, либо нечем доставить. В коротких случаях достаточно `track-progress` и одной строки в CLAUDE.md.

Отдельный случай — `/compact` внутри одной сессии. Compact частично сохраняет контекст сам, поэтому новая эстафета не нужна. Но если после compact модель «забыла» половину §2 (часто случается на длинных сессиях) — ручное чтение существующей эстафеты лечит ситуацию быстрее, чем переписывание.

## Что делает Claude Code при настройке этой практики

*(Этот раздел — инструкция для Claude Code на этапе прохождения чеклиста START.md, пункт 2.5.1. Пользователю читать необязательно.)*

### Уровень 0: базовая регистрация (по умолчанию)

Когда пользователь говорит «настроим эстафету» — **не создавай ни эстафетный файл, ни hook сразу**. Сначала добавь в CLAUDE.md помеченный блок про практику и обозначь триггеры её применения.

Минимальный блок в CLAUDE.md:

```markdown
<!-- [claude-code-practices v1.7] BEGIN: session-handoff -->
## Сеансовая эстафета

Перед длинной паузой (отпуск, передача проекта, конец дня крупной фазы)
или перед `/compact` сложного проекта — попроси меня собрать сеансовую
эстафету.

Команды активации (любая фраза по смыслу):
«давай сделаем хэндофф», «собери эстафету», «собери session handoff»,
«давай зафиксируем состояние перед паузой».

Я задам 5 вопросов (блокер, от кого ждём решение, сдвиги, запреты,
первая задача) и соберу самодостаточный документ для следующей сессии.

Хранить в `docs/SESSION-HANDOFF-YYYY-MM-DD.md`.
Подробности — `recipes/session-handoff.md`.
<!-- [claude-code-practices v1.7] END: session-handoff -->
```

Этот блок включает практику в проекте без создания пустого файла-заготовки. Пользователь обращается к ней, когда в реальности возникает повод.

### Уровень 1: первая эстафета по запросу

Если пользователь сразу говорит «давай попробуем составить первую эстафету» — спроси по очереди ключевые вопросы и заполни §0+§1, не больше:

1. «Что сейчас главный блокер? Одно предложение.»
2. «От кого ждёте решение, чтобы продолжить?»
3. «Какие 2–3 крупные сдвига закрыты за последний период?»
4. «Какие 2–3 запрета — что НЕ начинать без напоминания (например, "не подавать жалобу до экспертизы", "не мёрджить #142 до code review")?»
5. «Какая первая задача после прочтения этого файла?»

По ответам собери файл `docs/SESSION-HANDOFF-YYYY-MM-DD.md` с §0 и §1 по шаблону из рецепта. Остальные разделы (§2–§11) добавляй **только если пользователь просит расширить** — иначе оставляй файл компактным.

### Уровень 2: SessionStart hook (по запросу, после проверки окружения)

Если пользователь хочет автоматическую инжекцию эстафеты в новую сессию — настрой SessionStart hook в `.claude/settings.local.json`. Перед настройкой обязательно проверь:

1. **Есть ли уже SessionStart hook в settings?** Если да — добавь второй элемент в массив `SessionStart[0].hooks[]`, не заменяй.
2. **Установлен ли `jq`?** Запусти `jq --version`. Если нет — предупреди пользователя про установку (`choco install jq` для Windows, `brew install jq` для macOS) и предложи fallback (записать hook позже, после установки).
3. **Существует ли эстафетный файл хотя бы один?** Если нет — hook будет тихо ничего не выводить. Это нормально, но предупреди пользователя.
4. **Спроси про invariant-pointer LATEST.** Если пользователь согласен — hook читает `docs/SESSION-HANDOFF-LATEST.md`, при создании новой эстафеты обновляется только этот указатель. На Windows используй `cp` вместо симлинка (требует admin-прав).

После настройки — обязательно предложи **A/B-тест**: завершить сессию (`/exit`), запустить новую, задать контрольный вопрос. Без теста пользователь не узнает, что hook реально инжектирует контекст.

### Строгие правила (что НЕ делать)

1. **НЕ создавай эстафетный файл сразу при выборе пункта 2.5.1.** Базовый уровень — это блок в CLAUDE.md, не файл.
2. **НЕ настраивай hook без явного запроса пользователя.** Hook — это вмешательство в системные настройки, требует осознанного согласия.
3. **НЕ заполняй §0 за пользователя по своим догадкам.** Открытые вопросы, блокеры, грабли — это знание пользователя, не AI. Спрашивай.
4. **НЕ навязывай все 8 секций сразу.** Базовая первая эстафета — это §0+§1, остальное добавляется по мере роста проекта.
5. **НЕ создавай `docs/SESSION-HANDOFF-LATEST.md` если пользователь не выбрал invariant-pointer.** Достаточно файла с датой.
6. **НЕ путай эстафету и track-progress.** Если пользователь спрашивает «как сохранить, на чём остановился между сессиями за час» — это `track-progress`, не эстафета. Эстафета — для длинных пауз и фазовых переходов.

### Когда предлагать сразу настроить hook (а не отложить)

- Пользователь — разработчик, у него уже есть `.claude/settings.json` с другими hook'ами,
- Проект длинный, ожидаются недельные паузы,
- Пользователь сам говорит «хочу автоматически».

В остальных случаях — оставь hook на потом. Пользователь поймёт ценность практики после первой ручной эстафеты, и тогда сам попросит автоматизировать.

## Приложение: диагностика при отказе hook

Если hook не сработал (первый ответ чистой сессии не содержит фактов из §0):

1. **Файл существует?** `ls docs/SESSION-HANDOFF-LATEST.md` — должен быть. Если нет — создайте указатель: `cp docs/SESSION-HANDOFF-2026-04-14.md docs/SESSION-HANDOFF-LATEST.md`.
2. **Hook сохранился?** Прочитайте `.claude/settings.local.json` или `.claude/settings.json`, найдите блок `SessionStart`. Если отсутствует — настройка не применилась.
3. **Hook реально запускается?** Прямой вопрос модели: «был ли эстафетный файл в system-reminder при старте этой сессии?». Это отделяет «hook не сработал» от «hook сработал, но модель проигнорировала контекст».
4. **Размер ядра §0+§1 ≤10 000 символов?** `head -n 80 docs/SESSION-HANDOFF-LATEST.md | wc -c`. Если больше — сократите §0 или §1.
5. **`jq` установлен?** `jq --version`. Если нет — установите (см. раздел «Зависимости»).
6. **Команда работает в изоляции?** Скопируйте содержимое `command` из settings, запустите в терминале в корне проекта. Должен получиться валидный JSON.

Если все шесть шагов пройдены и поведение не воспроизводится — проблема скорее не в hook, а в системном окружении (версия Claude Code, права на файлы, конфликт с другим SessionStart-плагином).
