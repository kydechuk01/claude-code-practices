# TDD — тесты до кода (для разработчиков)

> Этот рецепт показывается, если ваша основная сфера — разработка. Для других сфер есть универсальный аналог: `recipes/contract-before-work.md`.

## Идея

Test-Driven Development (TDD) — подход, при котором сначала пишутся тесты, а потом код, который эти тесты проходит. С AI этот подход становится на порядок проще: Claude Code может сгенерировать сотню тестов за минуты, тогда как вручную это заняло бы дни.

TDD с AI — это не просто «хорошая практика». Это способ задать **контракт** системы до реализации: тесты определяют, что система должна делать, а код — как она это делает. Если тесты корректно описывают требования — код проверяется по ним автоматически.

## Цикл Red-Green-Refactor

Классический цикл TDD, который Claude Code выполняет за вас:

### 1. RED — напиши тест, который падает
```
Тест: «При вводе отрицательной суммы должна вернуться ошибка валидации»
Результат: FAIL (код ещё не написан)
```
Тест описывает ожидаемое поведение. Он должен падать — это подтверждает, что тест проверяет что-то реальное, а не проходит случайно.

### 2. GREEN — напиши минимальный код, чтобы тест прошёл
```
Код: if (amount < 0) throw new ValidationError('...')
Результат: PASS
```
Не пиши больше, чем нужно для прохождения теста. Минимальная реализация.

### 3. REFACTOR — улучши код, не ломая тесты
```
Рефакторинг: вынести валидацию в отдельный метод
Результат: все тесты по-прежнему PASS
```
Тесты — страховочная сетка. Они гарантируют, что рефакторинг ничего не сломал.

## Мультиагентный TDD — мощнейшая комбинация

Вместо одного агента, пишущего тесты, запустите трёх — каждый думает с позиции своей роли:

### Фаза 1: Генерация тестов (параллельно)
- **QA-инженер** — edge cases, граничные значения, негативные сценарии, нагрузка
- **Разработчик** — интеграционные тесты, контракты API, взаимодействие модулей
- **UX-дизайнер** — пользовательские сценарии, accessibility, отзывчивость UI

Каждый генерирует тесты независимо. Результат: значительный объём «сырых» тестов с разных ракурсов.

### Фаза 2: Синтез и дедупликация
Оркестратор (или вы сами) сводит результаты:
- Удаление дубликатов
- Группировка по модулям
- Приоритизация: критичные → важные → желательные

Результат: набор уникальных тестов с широким покрытием (количество зависит от сложности фичи).

### Фаза 3: Организация в файлы
Тесты раскладываются по файлам в соответствии со структурой проекта.

Результат: 10-15 тест-файлов с полным покрытием. **И всё это до написания единой строки реализации.**

## Как запустить TDD с Claude Code

### Простой вариант — один агент:

> Я хочу реализовать [фича]. Сначала напиши тесты, которые описывают ожидаемое поведение. Тесты должны падать. Потом напиши код, который проходит все тесты. Потом — рефакторинг.

### Мультиагентный вариант:

> Мне нужно реализовать [фича]. Организуй TDD с тремя агентами:
> 1. QA-инженер — edge cases, граничные значения, негативные сценарии
> 2. Разработчик — интеграционные тесты, контракты API
> 3. UX — пользовательские сценарии
>
> Каждый генерирует тесты независимо. Потом свести, дедуплицировать, организовать в файлы. Только после этого — реализация.

## Связь с Superpowers

Если у вас установлен плагин Superpowers, он включает встроенный навык TDD, который автоматически:
- Требует написать тест перед кодом (не даст писать код без теста)
- Проверяет, что тест действительно падает (RED)
- Проверяет, что тест проходит после реализации (GREEN)
- Напоминает про рефакторинг

Superpowers TDD + мультиагентная генерация тестов — максимальная комбинация.

## Когда TDD особенно важен

- **Сложная бизнес-логика** — расчёты, статусные машины, многошаговые процессы
- **API-контракты** — интеграция между сервисами
- **Рефакторинг** — тесты как страховка при изменении кода
- **Критичные системы** — финансы, медицина, безопасность

## Практический совет

Не пытайтесь покрыть тестами всё сразу. Начните с:
1. Основной happy path — «что система делает в нормальном случае»
2. Ключевые edge cases — «что происходит на границах»
3. Негативные сценарии — «что происходит при ошибке»

Claude Code сгенерирует остальное по мере реализации.
