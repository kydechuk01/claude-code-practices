# Инвентаризация проекта — поддержание порядка

## Проблема

Проект живёт: появляются новые файлы, удаляются старые, меняются решения. CLAUDE.md устаревает: ссылки ведут в никуда, упомянутые файлы не существуют, новые важные документы не упомянуты. Reference files расходятся с реальностью. AI работает с устаревшим контекстом — и выдаёт устаревшие решения.

Но у инвентаризации есть граница применимости, подтверждённая практикой: **она чинит механику (ссылки, реестры), но не лечит дублирование волатильных фактов.** На зрелом проекте инвентаризация проводилась регулярно и держала ссылки идеально (0 битых из 87) — а пересказ эстафеты в CLAUDE.md, добросовестно «актуализированный», протух снова за 8 дней. Копию нельзя поддерживать в актуальном состоянии — её можно только устранить (см. [`fact-ownership.md`](fact-ownership.md)).

## Решение: регулярная инвентаризация

Периодически (раз в неделю при активной работе, раз в месяц при спокойной) запускайте полную проверку проекта. Это занимает 5-10 минут, а экономит часы на будущих сессиях.

Если настроена самопроверка документации (раздел «Самопроверка» в [`fact-ownership.md`](fact-ownership.md)) — инвентаризация начинается с прогона скрипта: машинно-проверяемое ловится за минуту, глаза остаются для противоречий по смыслу.

## Промпт для инвентаризации

Скопируйте и вставьте в Claude Code:

```
Проведи инвентаризацию проекта:

0. Запусти bash scripts/doc-invariants.sh (если настроен) — почини все FAIL
   до зелёного. Дальше — то, что скриптом не ловится.
1. Аудит файлов — документы, данные, конфиги в рабочей директории;
   сироты вне структуры.
2. Проверка ссылок — CLAUDE.md, индекс памяти, рабочие доки → существующие
   файлы. И обратно: важные файлы без упоминаний.
3. Дубли волатильных фактов — по карте владельцев найди пересказы чужих
   фактов. УСТРАНИ дубль (замени ссылкой на владельца). ЗАПРЕЩЕНО
   актуализировать значение копии: актуализированный дубль протухнет снова.
4. Консистентность — противоречия между документами; правила,
   конфликтующие друг с другом.
5. Актуальность ловушек — каждую строку реестра в CLAUDE.md проверить:
   актуальна ли проблема? Устаревшие — пометить в карточке (_docs/gotchas/)
   как УСТАРЕЛА, убрать из реестра. Карточки не удалять.
6. Актуальность reference files — устарели ли справочные документы?
7. Зона покрытия — пройди ВСЕ поверхности: CLAUDE.md; эстафета + реестр
   архива; память (индекс + файлы, обе локации если их две); рабочие
   доки/методички; описания агентов; структура рабочих папок; корень.
8. Сравни с прошлым отчётом в audits/ — что закрыто, что висит, что новое.

Мандат: опечатки, записи индексов, реестры — чини сам; перемещения,
удаления, переименования файлов — только списком мне на решение.
Результат: отчёт в audits/audit-docs-YYYY-MM-DD.md + список вопросов.
```

## Ключевые отличия от наивной инвентаризации

- **Шаг 0 — машина прежде глаз.** Всё, что проверяется grep/wc, проверяет скрипт. Инвентаризация глазами регулярно пропускает то, что скрипт ловит первой строкой (переполненный индекс памяти, версии в чужих файлах).
- **Шаг 3 — устранять дубль, не актуализировать.** Раньше промпт говорил «обнови файл, актуализируй» — исполнитель добросовестно обновлял значение копии, и через неделю она врала снова. Инвентаризация в старой редакции — сборщик мусора для системы с протечкой: выгребает лужу, не закрывая кран.
- **Шаг 7 — явная зона покрытия.** Без списка поверхностей исполнитель проверяет лёгкое (ссылки) и пропускает больное (противоречия правил, описания агентов).
- **Шаг 8 — регресс-трекинг.** Отчёт каждой инвентаризации сохраняется в `audits/`; сравнение с предыдущим показывает, что закрыто, что висит, что новое. Без этого одни и те же находки переоткрываются.
- **Мандат.** Что чинится самостоятельно, а что — списком на решение оператора. Перемещение и удаление файлов — всегда решение человека.

## Что это даёт

- **Битые ссылки** — CLAUDE.md ссылается на `ref-architecture.md`, а файл переименован в `architecture-v2.md`. AI не находит его и работает без контекста.
- **Потерянные файлы** — за неделю появились новые спецификации, но CLAUDE.md о них не знает.
- **Устаревшие ловушки** — ловушка G-03 описывает проблему, которую уже починили.
- **Противоречия** — в одном файле «статусы заявки: 5», в другом — «7 статусов». AI путается.
- **Протухшие дубли** — четыре документа называют четыре разные «текущие фазы»; после шага 3 владелец остаётся один.

## Как внедрить

При настройке через START.md Claude Code добавит в ваш CLAUDE.md правило:

```markdown
## Обслуживание проекта

### Инвентаризация
Раз в неделю (при активной работе) или после крупных изменений — запросите:
«Проведи инвентаризацию проекта» — проверка инвариантов, аудит файлов,
ссылок, консистентности, устранение дублей волатильных фактов.
Отчёт — в audits/, сравнение с предыдущим.
```

Этого достаточно: Claude Code увидит правило и при необходимости напомнит о проверке. Либо вы сами запускаете промпт, когда чувствуете, что «что-то устарело».

## Когда запускать

- После крупного этапа работы (завершена фаза, сдан документ)
- Перед началом нового этапа (чтобы стартовать с чистым контекстом)
- Если AI начал давать странные ответы (возможно, работает с устаревшими данными)
- Раз в неделю как профилактика
