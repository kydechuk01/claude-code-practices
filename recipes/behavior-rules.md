# Правила поведения AI — как сделать Claude Code надёжным

## Зачем нужны правила

По умолчанию Claude Code старается быть полезным и соглашаться. Это хорошо для простых задач, но опасно для серьёзной работы:
- AI может выдумать факт, если не знает ответа (галлюцинация)
- AI может молча выполнить неоптимальное решение, хотя знает лучший вариант
- AI может сказать «готово», не проверив результат
- AI может предположить характеристику внешнего ресурса, вместо того чтобы проверить

Правила поведения — это строки в CLAUDE.md, которые меняют поведение Claude Code с «угодливого ассистента» на «ответственного специалиста».

## Основные правила

### 1. Документация прежде действий

**Правило для CLAUDE.md:**
```markdown
### Документация прежде действий
Если информации недостаточно — сначала прочитай документацию (reference files, _docs/, официальные доки), потом действуй. Не делать предположений на основе одного файла или фрагмента.
```

**Что это меняет:** Claude Code перестаёт угадывать и начинает проверять. Вместо «наверное, эта функция делает X» — открывает файл и читает.

**Почему важно:** без этого правила AI часто делает предположения на основе названия файла, функции или переменной. Предположение может быть неверным — и вся работа пойдёт по ложному пути.

### 2. Запрет на галлюцинации о внешних ресурсах

**Правило для CLAUDE.md:**
```markdown
### Факты о сторонних ресурсах
Любые утверждения о характеристиках сторонних сайтов, сервисов, продуктов, инструментов — подтверждать фактической проверкой (WebFetch, скриншот, документация). НЕ выдумывать и не предполагать на основе названия или категории.
```

**Что это меняет:** вместо «этот инструмент, скорее всего, поддерживает X» — Claude Code откроет документацию и проверит.

**Почему важно:** AI-модели уверенно называют несуществующие функции, неверные цены, устаревшие версии. Одно такое утверждение в отчёте — и вся работа теряет достоверность.

### 3. Ответственность — не соглашайся молча

**Правило для CLAUDE.md:**
```markdown
### Принцип ответственности
Если задача сформулирована некорректно, или предложенное решение не оптимально, или действие может привести к проблемам — скажи об этом. Предложи альтернативу с обоснованием. Необязательное согласие — фича: AI не должен молча делать то, что считает неправильным.
```

**Что это меняет:** Claude Code перестаёт быть «да-машиной». Если вы просите сделать X, а он видит, что Y будет лучше — он скажет: «Могу сделать X, но рекомендую Y, потому что...»

**Почему важно:** вы не всегда знаете все нюансы. Иногда AI видит проблему, которую вы не заметили. Молчаливое согласие — это не вежливость, это безответственность.

### 4. Отладка — сначала данные, потом гипотезы

**Правило для CLAUDE.md:**
```markdown
### Правило отладки
При ошибках ВСЕГДА начинать с данных:
1. Прочитать сообщение об ошибке целиком
2. Проверить логи / вывод / данные
3. Воспроизвести проблему
4. Только после этого — предлагать решение
НЕ угадывать причину. НЕ предлагать «попробуйте X» без понимания, почему X должен помочь.
```

**Что это меняет:** вместо серии попыток «а давайте попробуем это... а теперь вот это...» — Claude Code сначала разберётся в проблеме, а потом предложит одно точное решение.

**Почему важно:** случайные попытки часто маскируют проблему, а не решают её. Или создают новые проблемы.

### 5. Контроль качества вывода

**Правило для CLAUDE.md:**
```markdown
### Качество вывода
- Не пересказывать содержимое файлов — пользователь их видит
- Не комментировать каждое своё действие — просто делай
- Пояснения — пропорционально сложности: простое действие = без комментариев, нетривиальное решение = объяснение
- При работе с числами и данными — указывать источник каждой цифры
```

## Как добавить правила

Вставьте нужные правила в секцию «Обязательные правила» вашего CLAUDE.md. Claude Code прочитает их в начале каждой сессии.

Попросите Claude Code:

> Добавь в мой CLAUDE.md секцию «Обязательные правила» с правилами: документация прежде действий, запрет галлюцинаций о внешних ресурсах, принцип ответственности, правило отладки. Адаптируй формулировки под мой контекст.

## Правила для агентов

Если вы используете кастомных агентов (файлы в `~/.claude/agents/`), те же правила нужно прописать в файле агента. Агент — это отдельная роль, и он читает свой файл, а не CLAUDE.md основного проекта.

Ключевое правило для любого агента:

```markdown
# Правила
1. Если задача сформулирована некорректно — скажи об этом
2. Если знаешь лучший вариант — предложи его с обоснованием
3. Если данных недостаточно — запроси, а не додумывай
4. Каждое утверждение о внешних ресурсах — проверяй фактически
5. Перед ответом «готово» — проверь результат
```

## Результат

С этими правилами Claude Code ведёт себя как ответственный специалист:
- Проверяет факты, а не выдумывает
- Предлагает лучшие варианты, а не молча соглашается
- Ищет причину проблемы, а не пробует случайные решения
- Подтверждает результат, а не говорит «должно работать»
