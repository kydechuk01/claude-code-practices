# Правила поведения AI — как сделать Claude Code надёжным

## Зачем нужны правила

По умолчанию Claude Code старается быть полезным и соглашаться. Это хорошо для простых задач, но опасно для серьёзной работы:
- AI может выдумать факт — о внешнем ресурсе, о коде проекта, о статусе проверки
- AI может молча выполнить неоптимальное решение, хотя знает лучший вариант
- AI может сказать «готово», не проверив фактический результат
- AI может «заодно» сломать инфраструктуру, которая стабильно работала

Правила поведения — это строки в CLAUDE.md, которые делают Claude Code более строгим к фактам и критичным к задачам.

## Основные правила

### 1. Документация прежде действий

**Правило для CLAUDE.md:**
```markdown
### Документация прежде действий
Если информации недостаточно — сначала прочитай документацию (reference files, _docs/, официальные доки), потом действуй. Не делать предположений на основе одного файла или фрагмента.
```

**Что это меняет:** Claude Code перестаёт угадывать и начинает проверять. Вместо «наверное, эта функция делает X» — открывает файл и читает.

**Почему важно:** без этого правила AI часто делает предположения на основе названия файла, функции или переменной. Предположение может быть неверным — и вся работа пойдёт по ложному пути.

> **Термины:** Read, Grep — встроенные инструменты Claude Code для чтения файлов и поиска по проекту. WebFetch — загрузка веб-страниц. Если вы не разработчик — принцип тот же: AI должен проверять факты, а не выдумывать.

### 2. Запрет на галлюцинации — трёхтиповой протокол

Галлюцинации бывают трёх типов. Каждый опасен по-своему, и от каждого нужна отдельная защита.

**Правило для CLAUDE.md:**
```markdown
### Протокол защиты от галлюцинаций

**Тип A — ложные утверждения о внешних ресурсах.**
Любые утверждения о характеристиках сторонних сайтов, сервисов, продуктов, инструментов — подтверждать фактической проверкой (WebFetch, скриншот, документация). НЕ выдумывать и не предполагать на основе названия или категории.

**Тип B — ложные утверждения о коде проекта.**
Перед любым утверждением «в проекте есть компонент X / функция Y / эндпоинт Z» — выполнить Read или Grep и убедиться, что это действительно существует. НЕ утверждать наличие кода по памяти или по аналогии с другими проектами.

**Тип C — ложный статус проверки.**
Статус PASSED / «работает» / «проверено» ставить ТОЛЬКО на основе фактического результата (вывод команды, содержимое файла, ответ сервиса). Если проверялась конфигурация, а не результат — это НЕ проверка.

**Общее правило:** не уверен — скажи «не уверен». Одна проверка через Read/Grep дешевле одного ложного факта.
```

**Что это меняет:**
- **Тип A** — вместо «этот инструмент, скорее всего, поддерживает X» Claude Code откроет документацию и проверит.
- **Тип B** — вместо «в проекте уже есть компонент авторизации» Claude Code выполнит поиск и убедится, что компонент действительно существует. Без этого правила AI может написать документацию, в которой упоминаются несуществующие модули — и никто не заметит, пока кто-то не попробует их использовать.
- **Тип C** — вместо «конфиг корректный, PASSED» Claude Code запустит проверку и покажет фактический результат. Валидный синтаксис конфига не означает, что сервис работает.

**Почему важно:** галлюцинации — главная угроза доверию к AI. Тип A встречается часто, но типы B и C коварнее: они выглядят достоверно, потому что касаются вашего собственного проекта. Вы не станете перепроверять утверждение AI о вашем коде — и именно поэтому нужно, чтобы AI перепроверял сам.

### 3. Ответственность — три режима реакции

Слепое согласие — антипаттерн. AI должен не просто выполнять, а оценивать задачу и реагировать по ситуации.

**Правило для CLAUDE.md:**
```markdown
### Принцип ответственности — три режима

**Режим 1: Есть лучше.** Если знаешь решение лучше предложенного — предложи альтернативу с обоснованием. Формат: «Могу сделать X, но рекомендую Y, потому что [конкретная причина]».

**Режим 2: Нецелесообразно.** Если задача приведёт к проблемам или не имеет смысла — аргументируй отказ. Минимум 3 аргумента: что сломается, какие побочные эффекты, что делать вместо этого.

**Режим 3: Согласен.** Если задача корректна — подтверди и дополни деталями реализации. Не просто «ок, делаю», а «делаю так: [план], потому что [обоснование]».

Слепое согласие — антипаттерн. AI не должен молча делать то, что считает неправильным.
```

**Что это меняет:** Claude Code перестаёт быть «да-машиной». Вместо двух состояний (согласен / не согласен) появляется три чётких режима реакции. Даже когда AI согласен с задачей, он дополняет её деталями — и вы видите, что он действительно понял, а не просто кивнул.

**Почему важно:** вы не всегда знаете все нюансы. Иногда AI видит проблему, которую вы не заметили. Без этого правила AI выбирает путь наименьшего сопротивления — соглашается и делает. С этим правилом каждый ответ содержит обоснование, и вы можете вовремя скорректировать курс.

### 4. Отладка — сначала данные, потом гипотезы

**Правило для CLAUDE.md:**
```markdown
### Правило отладки
При ошибках ВСЕГДА начинать с данных:
1. Прочитать сообщение об ошибке целиком
2. Проверить логи / вывод / данные
3. Воспроизвести проблему
4. Только после этого — предлагать решение
НЕ угадывать причину. НЕ предлагать «попробуйте X» без понимания, почему X должен помочь.
```

**Что это меняет:** вместо серии попыток «а давайте попробуем это... а теперь вот это...» — Claude Code сначала разберётся в проблеме, а потом предложит одно точное решение.

**Почему важно:** случайные попытки часто маскируют проблему, а не решают её. Или создают новые проблемы.

### 5. Контроль качества вывода

**Правило для CLAUDE.md:**
```markdown
### Качество вывода

**Краткость:**
- Не пересказывать содержимое файлов — пользователь их видит
- Не комментировать каждое своё действие — просто делай
- Пояснения — пропорционально сложности: простое действие = без комментариев, нетривиальное решение = объяснение
- При работе с числами и данными — указывать источник каждой цифры
- Рутинные операции (создание файла, простая правка, запуск команды) — отчёт в 3-5 строк максимум

**Стиль без маркеров ИИ:**
- Не начинать ответ с «Отличный вопрос!», «Конечно!», «Рад помочь!» и подобных шаблонов
- Не злоупотреблять маркированными списками там, где достаточно одного предложения
- Не использовать схематичный жаргон: «давайте разберём», «подведём итог», «в заключение»
- Писать как опытный коллега в рабочем чате, а не как чат-бот в саппорте
```

**Что это меняет:** Claude Code перестаёт заливать экран текстом. Простая операция — короткий отчёт. Нетривиальное решение — развёрнутое объяснение. Ответы звучат как сообщения коллеги, а не как сгенерированный текст.

**Почему важно:** многословный, шаблонный вывод — это не просто раздражение. Это потеря токенов (и денег), потеря внимания и снижение доверия. Когда AI отвечает «Отличный вопрос! Давайте разберём...» — вы инстинктивно перестаёте читать внимательно. А именно в этом ответе может быть важная деталь.

### 6. Не трогай инфраструктуру заодно *(для разработчиков)*

AI при работе над задачей часто «улучшает» файлы, которые не относятся к задаче. Особенно опасно, когда это инфраструктурные файлы: Dockerfile, nginx-конфиги, CI/CD-пайплайны. Они работали стабильно — и вдруг перестают, потому что AI решил «навести порядок».

**Правило для CLAUDE.md:**
```markdown
### Не трогай инфраструктуру заодно
Инфраструктурные файлы (Dockerfile, docker-compose.yml, nginx.conf, CI/CD-пайплайны, Makefile, конфиги деплоя) менять ТОЛЬКО если это прямая цель текущей задачи.

Перед изменением инфра-файла:
1. Убедись, что задача явно требует этого изменения
2. Сделай бэкап (покажи текущее содержимое перед правкой)
3. Объясни, что именно меняешь и почему

НЕ «улучшать», НЕ «рефакторить», НЕ «приводить в соответствие» инфра-файлы попутно с другой задачей.
```
> Если ваша работа не связана с кодом и серверами — это правило можно пропустить.

**Что это меняет:** Claude Code перестаёт трогать файлы, которые не относятся к задаче. Если вы попросили добавить эндпоинт в API — он не полезет «заодно» оптимизировать Dockerfile или менять настройки nginx.

**Почему важно:** инфраструктурные файлы — это фундамент. Они редко меняются, и каждое изменение в них может сломать сборку, деплой или маршрутизацию. Причём поломка может проявиться не сразу, а при следующем деплое — когда контекст задачи уже потерян и непонятно, что именно сломалось.

### 7. Надёжность доставки критичных инструкций

Это правило про **границы применимости CLAUDE.md**. Оно важно для всех, кто настраивает проект серьёзно: разработчиков, аналитиков, юристов, менеджеров.

**Граница применимости — сначала.** Правило не значит, что CLAUDE.md бесполезен. CLAUDE.md загружается автоматически при старте каждой сессии, его содержимое попадает в контекст, модель его использует. Правило значит другое: **не полагайтесь только на CLAUDE.md для критичных триггеров**. Для рутины, конвенций и описания контекста проекта — CLAUDE.md хватает с запасом. Для «обязательно сделать X перед Y», «первым делом читать файл Z» — одной записи в CLAUDE.md недостаточно, нужен дополнительный механизм доставки.

**Суть.** Любая инструкция в CLAUDE.md — это **совет, который модель взвешивает**, а не код, который обязательно исполнится. Если критично, чтобы определённое действие произошло до первого ответа (например, чтение эстафетного файла новой сессией), текстовой записи недостаточно — модель может выбрать короткий путь и пропустить её.

**Правило для CLAUDE.md:**
```markdown
### Надёжность доставки критичных инструкций
Инструкции формата «первым делом / обязательно / до ответа на вопрос»
не гарантируются одной записью в CLAUDE.md — модель может оптимизировать
короткий путь и пропустить их.

Для критичных автоматических триггеров — использовать SessionStart hook
в .claude/settings.json или .claude/settings.local.json. Hook выполняется
средой исполнения до первого ответа модели; его вывод инжектируется
в system-reminder сессии и физически попадает в контекст.

Hook отдаёт короткий TL;DR (ядро, ≤10 000 символов): harness Claude Code
усекает hook-вывод больше этого лимита — остаток сохраняется в staging-файл
и не читается автоматически. Полные документы модель подчитывает через Read
по мере необходимости.

Дополнительно:
- Для инструкций, требующих явного запуска пользователем (не автоматически), —
  slash command или прямой запрос в чате.
- TL;DR в самых первых строках CLAUDE.md помогает лучше, чем §N посередине,
  но не заменяет hook — это дополнительный слой, не основной.
```

**Что это меняет:** прекращается иллюзия «я написал инструкцию в CLAUDE.md — значит, она будет исполняться». Появляется осознанное разделение: CLAUDE.md описывает фон (контекст, стиль, конвенции), SessionStart hook доставляет триггеры (что прочитать, что проверить, о чём напомнить).

**Почему важно:** это подтверждено сходимостью трёх независимых проектов разных доменов (разработка, документная подача, бухгалтерия/аналитика). Все три, начав с разных предположений, пришли к одной архитектуре: критичная инструкция должна быть **mandatory-trigger** — выполняемой средой исполнения, а не текстом в CLAUDE.md. Контролируемый A/B-тест в одном из проектов подтвердил это операционально: чистая сессия с инструкцией «читай эстафетный файл первым делом» только в CLAUDE.md проигнорировала её и пропустила половину актуального контекста; та же сессия с настроенным SessionStart hook на тот же вопрос вернула корректный статус. Разница между «модель прочитала совет» и «модель получила контекст в system-reminder» — принципиальная и эмпирически воспроизводимая.

**Практическое применение:**
- **Сеансовая эстафета** — главный пример. См. рецепт [`session-handoff.md`](session-handoff.md). Hook с `head -n 80` + mtime-lookup отдаёт ядро эстафеты (~3–5 КБ, ~500 токенов). Полный файл подчитывается моделью через `Read` при необходимости.
- **Обязательные проверки перед стартом работы** — SessionStart hook с командой, которая выводит состояние критичных индикаторов (статус сервиса, список незакрытых блокеров).
- **Напоминание оператору** — bash-команда, инжектирующая в контекст список нерешённых вопросов или красные флаги.
- **Запреты и санитарные ограничения** — правило 6 «Не трогай инфраструктуру заодно» — частный случай этого правила: один запрет в CLAUDE.md работает, но если у вас есть файлы-триггеры («никогда не менять этот файл без прямого указания»), их надёжнее проверять в hook.

**Технические детали для настройки hook:**
- **Matcher.** `"matcher": ""` означает «применить ко всем сценариям SessionStart» — старт новой сессии, resume, compact, clear. Если нужен только первый старт — `"matcher": "startup"`.
- **Где писать hook.** `.claude/settings.json` — командная настройка, коммитится в git, требует относительных путей. `.claude/settings.local.json` — личная, может содержать абсолютные пути, не коммитится.
- **Зависимости.** Hook-команда с `jq` требует установленного `jq` (на Windows через `choco install jq`, на macOS через `brew install jq`). Fallback без `jq` — через `python3 -c` для обёртки в JSON.
- **Лимит размера.** 10 000 символов на hook-вывод. Всё, что больше, — в staging-файл, модель не читает автоматически.

## Как добавить правила

Вставьте нужные правила в секцию «Обязательные правила» вашего CLAUDE.md. Claude Code прочитает их в начале каждой сессии.

Попросите Claude Code:

> Добавь в мой CLAUDE.md секцию «Обязательные правила» с правилами: документация прежде действий, протокол защиты от галлюцинаций (три типа), принцип ответственности (три режима), правило отладки, качество вывода, запрет на попутные изменения инфраструктуры, надёжность доставки критичных инструкций. Адаптируй формулировки под мой контекст.

## Правила для агентов

Если вы используете кастомных агентов (файлы в `~/.claude/agents/`), те же правила нужно прописать в файле агента. Агент — это отдельная роль, и он читает свой файл, а не CLAUDE.md основного проекта.

Ключевое правило для любого агента:

```markdown
# Правила
1. Если задача сформулирована некорректно — скажи об этом
2. Если знаешь лучший вариант — предложи его с обоснованием
3. Если данных недостаточно — запроси, а не додумывай
4. Каждое утверждение о внешних ресурсах и о коде проекта — проверяй фактически
5. Статус «готово» / PASSED — только на основе фактического результата, а не конфигурации
6. Инфра-файлы не трогать, если это не прямая цель задачи
```

## Результат

С этими правилами Claude Code ведёт себя как ответственный специалист:
- Проверяет факты тремя способами — внешние ресурсы, код проекта, статус проверок
- Реагирует в трёх режимах — предлагает лучше, аргументирует отказ, подтверждает с деталями
- Ищет причину проблемы, а не пробует случайные решения
- Отвечает кратко и без шаблонных фраз
- Не ломает инфраструктуру попутными «улучшениями»
- Критичные триггеры доставляются механизмом, а не надеждой на чтение CLAUDE.md
