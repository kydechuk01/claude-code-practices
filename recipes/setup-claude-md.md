# Как написать CLAUDE.md

## Что это

Файл `CLAUDE.md` в корне рабочей директории — автоматически загружается в каждую сессию Claude Code. Без него AI не знает контекст вашей работы. С ним — сразу понимает, с чем имеет дело.

Без CLAUDE.md каждый разговор начинается с нуля: AI не знает, чем вы занимаетесь. С CLAUDE.md он сразу в контексте.

## Структура

### Обязательные секции

**Описание проекта/области работы** — 1-2 предложения. Claude Code читает их первыми и сразу понимает контекст:
```markdown
# [Название]
Анализ финансовой отчётности группы компаний. Данные в SQLite, отчёты в markdown и PDF.
```

**Контекст** — инструменты, источники данных, терминология. Чем подробнее — тем точнее AI работает с вашими данными:
```markdown
## Контекст
- Данные: SQLite база, таблицы invoices, payments, contracts
- Отчёты: markdown → PDF через Playwright
- Период: Q1-Q4 2025
- Валюта: RUB, USD (курс ЦБ на дату операции)
```

**Конвенции** — как оформлять результаты, именовать файлы. Без этого AI будет каждый раз делать по-разному:
```markdown
## Конвенции
- Файлы: латиница, kebab-case (report-q1-2025.md)
- Содержимое: русский язык
- Таблицы: суммы с разделителем тысяч, 2 знака после запятой
- Даты: DD.MM.YYYY в отчётах, YYYY-MM-DD в именах файлов
```

**Обязательные правила** — правила поведения AI (подробнее в `recipes/behavior-rules.md`):
```markdown
## Обязательные правила
- Документация прежде действий — сначала прочитай, потом делай
- Факты о внешних ресурсах — подтверждать проверкой, не выдумывать
- Принцип ответственности — если знаешь лучший вариант, скажи
```

**Известные ловушки** — проблемы, на которые вы уже потратили время. Это самая ценная секция: каждая запись экономит будущие часы отладки. Два уровня: компактный реестр в CLAUDE.md (до 20 строк) и подробные карточки в `_docs/gotchas/`:
```markdown
## Известные ловушки

| # | Ловушка | Решение | Дата |
|-|-|-|-|
| G-01 | Кодировка CSV из внешних систем — Windows-1251 вместо UTF-8 | Конвертировать при импорте | 2026-01-10 |
| G-02 | Выгрузка за январь содержит корректировки декабря | Фильтровать по дате документа | 2026-01-15 |

Подробности (причина, предотвращение): `_docs/gotchas/`
```

### Рекомендуемые секции

**Текущие задачи** — указатель на текущий этап работы. Позволяет Claude Code после любого перерыва продолжить с того места, где остановились (подробнее в `recipes/track-progress.md`):
```markdown
## Текущие задачи
**Текущий шаг:** Фаза 2 → анализ данных за Q3
→ Оперативный трекер: `current-sprint.md`
→ Полный план: `global-plan.md` (12 задач, 5 выполнено)
```

**Маппинг агентов** — какой агент для какой задачи.

**Ссылки на reference files** — где искать подробности:
```markdown
## Reference files
- `ref/schema.md` — описание структуры данных
- `ref/glossary.md` — терминология и сокращения
```

## Размеры в реальных проектах

- Простая задача: ~50 строк
- Средний проект: ~200 строк
- Сложный домен (сетевое оборудование, инфраструктура): ~400 строк

## Принцип

В CLAUDE.md — только то, что нужно в **каждой** сессии. Подробности — в reference files. Не превращайте CLAUDE.md в энциклопедию на 1000 строк — это забьёт контекстное окно AI и снизит качество работы.

## Известные ловушки — главная ценность

Каждая ловушка — проблема, на которую уже потрачено время. Без этих записей AI наступает на те же грабли. С ними — обходит проблему автоматически.

### Протокол пополнения

1. Проблема заняла >10 минут или повторилась → создать карточку в `_docs/gotchas/G-NN.md`
2. Добавить строку в таблицу «Известные ловушки» в CLAUDE.md
3. Нумерация сквозная: G-01, G-02... (без пропусков)

### Формат карточки

```markdown
# G-NN: [заголовок]
Дата: YYYY-MM-DD | Критичность: высокая/средняя/низкая | Область: [домен]

## Проблема
Что произошло.

## Причина
Почему это произошло.

## Решение
Как починили.

## Предотвращение
Что сделать, чтобы не повторилось.
```

### Когда ловушка становится правилом

Если ловушка порождает общее правило (например, «всегда проверять кодировку при импорте») — добавить правило в секцию «Обязательные правила» CLAUDE.md. В карточке пометить: `Перенесено в правила`.

### Интеграция в агентов

Агенты, работающие с областью ловушки, должны читать соответствующие карточки перед выполнением задачи. В файле агента добавить:
```
Перед началом работы прочитай карточки ловушек: _docs/gotchas/G-*.md
```

### Ревизия

При инвентаризации проекта (`recipes/project-inventory.md`) — проверить каждую строку реестра: актуальна ли проблема? Устаревшие — пометить в карточке как УСТАРЕЛА, убрать из реестра. Карточки не удалять (история).

### Лимит реестра

Реестр в CLAUDE.md — до 20 строк. Если больше — вынести редкие ловушки из реестра, оставив только карточки. В реестре держать самые частые и критичные.

## Если CLAUDE.md уже существует

Если в проекте уже есть CLAUDE.md:
- **НЕ перезаписывать**
- Показать текущее содержимое пользователю
- Предложить добавить недостающие секции (правила поведения, известные ловушки, текущие задачи)
- Каждое изменение — с подтверждением пользователя
- Перед изменением создать бэкап: `CLAUDE.md.bak`

## Как создать с нуля

Попросите Claude Code:

> Проанализируй файлы в текущей директории и создай CLAUDE.md. Включи: описание, контекст (инструменты, данные, терминология), конвенции оформления, обязательные правила поведения. Секцию «Известные ловушки» оставь пустой — буду заполнять по мере работы. Добавь секцию «Текущие задачи» с указателем на текущий этап.

Если рабочая директория пуста (новый проект) — не анализируй файлы, а спроси пользователя, что за проект планируется, и создай CLAUDE.md на основе его рассказа.

Каждую секцию, добавленную пакетом, оборачивай метками:
```
<!-- [claude-code-practices v1.0] BEGIN: название секции -->
...содержимое...
<!-- [claude-code-practices v1.0] END: название секции -->
```

Это позволяет при обновлении или удалении точно определить, какие блоки добавлены пакетом, а какие написаны пользователем. Пользовательские секции НИКОГДА не помечаются и не удаляются.

**Важно:** предупреди пользователя, что содержимое между метками BEGIN/END при обновлении или удалении пакета будет заменено или удалено целиком. Если пользователь хочет добавить свои правки — лучше делать это ВНЕ меченых блоков.

### Если метки не найдены

Если при повторном запуске метки BEGIN/END не обнаружены (пользователь удалил или отредактировал их):

1. Ищи по заголовкам секций: «## Обязательные правила», «## Известные ловушки», «## Текущие задачи», «## Контекст», «## Конвенции»
2. Если заголовок найден — покажи содержимое и спроси: обновить и восстановить метки, оставить как есть, или добавить новую секцию рядом?
3. Если заголовок НЕ найден — добавь секцию как новую (с метками)
4. Никогда не добавляй секцию с тем же заголовком, что уже есть — это создаст дубли

Claude Code создаст первую версию за один проход. Дальше — дополняйте по мере работы.

## Статус настройки пакета

При первом прохождении чеклиста добавь в CLAUDE.md блок с текущим статусом:

```markdown
<!-- [claude-code-practices v1.0] BEGIN: setup-status -->
## Настройка Claude Code Practices

Статус: В ПРОЦЕССЕ
Пройдено: уровень 1 (пункты 1.1, 1.2, 1.4)
Отложено: голосовой ввод (1.3)
Не пройдено: уровень 2, уровень 3
Путь к пакету: [путь к START.md]

Чтобы продолжить настройку — скажи:
«Прочитай файл [путь] и продолжи настройку с того места, где остановились»
<!-- [claude-code-practices v1.0] END: setup-status -->
```

Этот блок позволяет Claude Code в следующей сессии увидеть, что настройка не завершена, и предложить продолжить. Когда все пункты пройдены — обнови статус на «ЗАВЕРШЕНО».

## Обновление между версиями пакета

При повторном запуске START.md с новой версией пакета:

1. Найди метки в CLAUDE.md — определи версию (например, `v1.0`)
2. Сравни с текущей версией пакета (из VERSION.md)
3. Если версия CLAUDE.md старше — покажи пользователю, какие секции изменились
4. Обнови помеченные блоки с подтверждением пользователя, пользовательские секции не трогай
5. Обнови номер версии в метках
