# Один факт — один владелец

## Как вызвать

Скажите Claude Code в обычном чате:

> «составь карту владельцев фактов» · «проверь дубли изменчивых фактов» · «где у нас живёт текущий статус?» · «документация не врёт?»

Точная формулировка не важна. Claude Code построит карту по реальной структуре вашего проекта и найдёт пересказы, которые пора заменить ссылками.

## Главный принцип

**Копия не бывает актуальной: либо ссылка, либо дефект.** Каждый изменчивый факт живёт ровно в одном документе. Остальные документы — только ссылка на владельца, без пересказа значения.

Житейский пример: дата заседания живёт в календаре дел. В записке по делу — не дата, а ссылка на календарь: перенесли заседание — записка не соврёт.

Изменчивый (волатильный) факт — то, что меняется чаще, чем раз в месяц: версия, статус задачи, текущая фаза, реквизиты окружения, суммы за период. Стабильный факт — конвенции, правила, термины, шаблоны — живёт где удобно, на него правило не распространяется.

## Болезнь, которую это лечит

Пересказ изменчивого факта в «стабильном» документе протухает за дни, потому что владелец факта обновляется чаще, чем его копии. Это не проблема дисциплины — это проблема архитектуры документации: копию нельзя «поддерживать в актуальном состоянии», её можно только устранить.

Болезнь подтверждена на пяти независимых контекстах из четырёх доменов:

- **Проект тестирования корпоративной BPM-системы:** через 3 месяца работы по практикам — ~80 дефектов документации, все критичные имели одну природу. Пересказ эстафеты в CLAUDE.md «актуализировали» — он протух снова за 8 дней. На одну дату четыре документа называли четыре разные «текущие фазы».
- **Проект веб-разработки:** статусный файл с фразой «версия при следующем билде: N» протух за 9 часов — версия была собрана и выехала в прод в тот же день.
- **Проект бухгалтерской аналитики:** оперативный трекер отстал от эстафеты на 4 дня и содержал устаревшую сумму налога (расхождение ~10%); налоговая ставка, захардкоженная в трёх агентах, устарела при смене регионального закона — агенты продолжали советовать по старой ставке.
- **Сам этот пакет практик:** строка «Версия N» в CLAUDE.md дублировала VERSION.md — рассинхрон чинился вручную при каждом релизе.
- **Адаптация пакета под другую платформу:** тот же рассинхрон версии пойман автоматической проверкой дважды за неделю.

Общее во всех пяти: **копию добросовестно обновляли — и она протухала снова.** Единственное лечение — устранить копию.

## Карта владельцев фактов

Решение — таблица в CLAUDE.md, которая закрепляет: какой класс фактов где живёт. Универсальный скелет (адаптируется под ваш домен при настройке):

| Класс фактов | Типичный владелец |
|---|---|
| Текущая фаза, статусы задач, очередь, правила периода | эстафетный файл (session handoff) |
| Реестр версий эстафет | INDEX-файл архива |
| Окружение: доступы, адреса, версии систем | reference-файлы памяти |
| Предметные материалы: досье, методички, процедуры | профильный документ или папка в docs/ |
| Стабильные правила, конвенции, журнал ловушек | CLAUDE.md |
| Маркер «последний шаг», статус установки пакета | CLAUDE.md |
| Уроки решённых задач | файлы памяти (агенты ссылаются, не копируют) |
| Указатели «что где лежит» | индекс памяти — одна строка на запись |

Сама карта — про **классы** фактов, не про значения: она меняется только при смене структуры проекта, то есть по определению выше — стабильный факт. Её владелец — CLAUDE.md.

### Важный нюанс: владельцем может быть и CLAUDE.md

Правило — «один владелец», а не «CLAUDE.md только для стабильного». Маркер «Последний шаг» из рецепта [`track-progress.md`](track-progress.md) — изменчивый, но живёт **только** в CLAUDE.md: CLAUDE.md и есть его владелец, других носителей нет, нарушения нет. То же со статусом установки пакета и журналом ловушек.

Нарушение — когда факт живёт в двух местах. Версия эстафеты, записанная и в эстафете, и в CLAUDE.md — дефект, потому что у эстафеты уже есть владелец.

## Чек-листы на событиях-источниках

У каждого повторяющегося дефекта документации есть **событие-источник**. К событию прикрепляется чек-лист из 3–4 строк — и дефект не возникает. Это дешевле любой периодической уборки (правило 8 в [`behavior-rules.md`](behavior-rules.md)).

- **Новая версия эстафеты** → чек-лист в разделе «Жизненный цикл эстафет» рецепта [`session-handoff.md`](session-handoff.md) — он владелец этого чек-листа.
- **Закрытие задачи** → итоговый документ в папку задачи; урок — в файл памяти, агенту ссылка; статус попадёт в эстафету при ближайшей новой версии — больше никуда.
- **Создан регулярно обновляемый файл** (трекер, журнал, статусник) → строка в карту владельцев: каким классом фактов он владеет. Файл, которого нет в карте, никем не обновляется по событию — и тихо протухает, оставаясь «авторитетным» на вид.

## Анти-паттерны

**«Снимок эстафеты в CLAUDE.md».** Пересказ текущей фазы, версий, статусов из эстафеты в CLAUDE.md «для удобства». Эмпирика: пережил 4 версии эстафеты и дезинформировал старт каждой сессии 8 дней подряд. Правильно: в CLAUDE.md — только триггер «первым делом читай эстафету» (а лучше — механизм доставки из правила 7). Положительный пример: проект, где CLAUDE.md содержал **только указатель** на LATEST-файл эстафеты, через месяц интенсивной работы остался чист — указатель не протухает, потому что не содержит значения.

**«Индекс памяти как второй носитель знаний».** Записи-простыни в индексе дублируют тела файлов памяти и устаревают отдельно от них. Правильно: запись индекса — указатель «что это + когда открывать», одна строка (см. [`setup-memory.md`](setup-memory.md)).

**«Факт среды захардкожен в агенте».** Адрес стенда, ставка налога, версия системы — внутри описания агента. Протухает при первой переустановке среды или смене закона; агент продолжает уверенно советовать по устаревшим данным. В регуляторных доменах (налоги, право) это уже не косметика, а неверные рекомендации. Правильно: «все реквизиты — в reference_X, сюда не копировать» (см. [`create-agents.md`](create-agents.md)).

## Самопроверка документации

**Что можно проверить машинно — не должно проверяться глазами.** Инвариант — проверяемое утверждение о документации: «в CLAUDE.md нет номеров версий эстафеты», «каждый файл архива есть в реестре», «индекс памяти меньше лимита автозагрузки».

Эмпирика: на зрелом проекте инвентаризация глазами проводилась регулярно и идеально чинила ссылки (0 битых из 87) — но не заметила, что индекс памяти превысил лимит автозагрузки и 8 последних записей **молча не попадали ни в одну сессию**. Машинная проверка ловит такое первой же строкой.

### Для всех: проверка по запросу

Скажите «документация не врёт?» или «проверь дубли по карте владельцев» — Claude Code пройдёт по карте и найдёт нарушения вручную. Скрипт не нужен.

### Для разработчиков: генерация скрипта

Claude Code генерирует `scripts/doc-invariants.sh` из карты владельцев вашего проекта. Пять универсальных инвариантов-каркасов:

1. **Индекс памяти меньше лимита автозагрузки.** Документированное поведение Claude Code: загружаются первые **200 строк или первые 25 KB** файла MEMORY.md — что наступит раньше; хвост молча не загружается. Для кириллицы в UTF-8 байтовый лимит достигается раньше символьного (~1.5–2 байта на символ).
2. **В CLAUDE.md нет изменчивых значений чужих владельцев** — паттерны из карты: номера версий эстафет, словарь статусов. *Опциональный strict-режим:* словарь статусов («в работе», «заблокировано») встречается и в обычной прозе — этот инвариант может ложно срабатывать; включайте его осознанно или сужайте паттерны.
3. **Реестр архива эстафет полон:** каждый файл архива упомянут в INDEX, текущая версия зарегистрирована.
4. **Ссылки индекса памяти резолвятся** — ноль битых.
5. **Записи индекса не разрастаются** — порог длины (~200 символов), мягкий лимит на исключения.

Каркас скрипта стабилен:

```bash
#!/usr/bin/env bash
# Инварианты документации. Сгенерировано из карты владельцев <дата>.
# Лимиты (200 строк / 25 KB) — поведение Claude Code на момент генерации;
# при странностях перепроверьте актуальную документацию.
cd "$(dirname "$0")/.." || exit 1
fail=0
chk() { if [ "$2" -eq 0 ]; then echo "OK   $1"; else echo "FAIL $1"; fail=1; fi; }

# Шаблоны проверок (генерируются из карты владельцев проекта):
# [размер]   size=$(wc -c < "<индекс памяти>"); [ "$size" -lt 25000 ]; chk "..." $?
# [чистота]  ! grep -qE "<паттерн версий>" CLAUDE.md; chk "..." $?
# [реестр]   for f in <архив>/*.md; do grep -q "$(basename "$f")" <INDEX> || ...; done
# [ссылки]   for f in $(grep -oE "\]\([a-z0-9_-]+\.md\)" <индекс> | ...); do [ -f ... ]
# [длина]    long=$(awk 'length($0) > <порог> && /^- \[/' <индекс> | wc -l); [ "$long" -le <N> ]

echo "---"
[ "$fail" -eq 0 ] && echo "ВСЕ ИНВАРИАНТЫ ЗЕЛЁНЫЕ" || echo "ЕСТЬ НАРУШЕНИЯ"
exit $fail
```

Скрипт — bash: нужен Git Bash или WSL. Если bash недоступен — попросите PowerShell-аналог с той же логикой.

**Когда запускать:** в чек-листе новой версии эстафеты; шагом 0 инвентаризации ([`project-inventory.md`](project-inventory.md)); когда кажется, что документация врёт.

## Блок для CLAUDE.md

При настройке Claude Code добавляет в ваш CLAUDE.md блок ниже. Плейсхолдеры в угловых скобках заполняются реальными путями проекта; `vN.N` в метках — версия пакета, в которой блок установлен (по меткам будущие обновления находят блоки пакета).

```markdown
<!-- [claude-code-practices vN.N] BEGIN: fact-ownership -->
## Карта владельцев фактов (single source of truth)

Каждый изменчивый факт живёт ровно в ОДНОМ документе. Остальные — только
ссылка на владельца, без пересказа значения. Пересказ = дефект документации
(находится самопроверкой scripts/doc-invariants.sh и инвентаризацией).

| Класс фактов | Единственный владелец |
|-|-|
| Текущая фаза, статусы задач, очередь | <эстафетный файл> |
| Реестр версий эстафет | <INDEX архива> |
| Окружение: доступы, адреса, версии систем | <reference-файлы памяти> |
| Стабильные правила, конвенции, ловушки | CLAUDE.md (этот файл) |
| Указатели «что где лежит» | <индекс памяти> — 1 строка ≤200 симв. |

При новой версии эстафеты — чек-лист из session-handoff (архив, INDEX,
факты ссылками, самопроверка). При закрытии задачи — итог в папку задачи,
урок в память, статус только в эстафету.
<!-- [claude-code-practices vN.N] END: fact-ownership -->
```

## Что делает Claude Code при настройке этой практики

*(Инструкция для Claude Code на этапе чеклиста START.md. Пользователю читать необязательно.)*

### Уровень 0: карта владельцев (по умолчанию)

1. Изучи реальную структуру проекта: где эстафета, где память, где трекеры, где агенты.
2. Построй карту владельцев из скелета выше, заполнив плейсхолдеры реальными путями. Классы фактов, которых в проекте нет (например, нет эстафеты), — убери из таблицы.
3. Покажи черновик пользователю, спроси подтверждение.
4. Вставь блок с метками в CLAUDE.md, подставив текущую версию пакета из VERSION.md.

### Уровень 1: чистка существующих дублей (по запросу)

1. По карте владельцев найди пересказы чужих фактов: grep по характерным паттернам (номера версий, словарь статусов, суммы) во всех документах, кроме владельца.
2. Покажи список найденных дублей пользователю.
3. После подтверждения — замени каждый дубль ссылкой на владельца. **ЗАПРЕЩЕНО обновлять значение копии** — актуализированный дубль протухнет снова.

### Уровень 2: генерация скрипта самопроверки (по запросу, для разработчиков)

1. **Не генерируй без карты владельцев** — инварианты выводятся из неё.
2. Сгенерируй `scripts/doc-invariants.sh` из карты по каркасу выше.
3. **Каждый инвариант прогони и покажи зелёным до фиксации.** Ложно срабатывающий инвариант хуже отсутствующего: пользователь перестанет доверять скрипту. Инвариант №2 (словарь статусов) предлагай как опциональный.
4. На Windows проверь наличие bash (`bash --version`); если нет — предложи PowerShell-вариант.

### Строгие правила (что НЕ делать)

1. **НЕ строй карту по скелету вслепую** — сначала изучи реальную структуру проекта.
2. **НЕ удаляй найденные дубли без показа пользователю** — в «дубле» может оказаться осознанное исключение.
3. **НЕ актуализируй значения копий** — никогда. Либо устранить дубль, либо оставить с пометкой и вопросом к пользователю.
4. **НЕ распространяй правило на стабильные факты** — конвенции, термины и шаблоны могут повторяться, это нормально.
5. **НЕ генерируй скрипт без карты и НЕ зашивай ложно-срабатывающие паттерны.**
