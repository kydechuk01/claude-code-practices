# Параллельные агенты с watchdog *(для разработчиков)*

## О чём этот рецепт

Методика запуска нескольких фоновых агентов с автоматическим обнаружением зависших и безопасным перезапуском. Дополняет базовый рецепт [multi-agent-collaboration.md](multi-agent-collaboration.md): тот описывает **семантику** коллегиальной работы (роли, фазы, синтез), а этот — **операционный failover**, что делать когда агент завис.

**Аудитория:** разработчики, знакомые с bash/Python/JSON. Если вы не разработчик — задайте задачу Claude Code обычным запросом про мультиагентную работу, оркестратор сам выберет подходящий механизм. Ручная реализация watchdog не требуется для большинства сценариев.

---

## Честный disclaimer

В большинстве случаев встроенных механизмов Claude Code достаточно: фоновые агенты отправляют уведомление о завершении, и оркестратор получает результат. Watchdog — это **страховочный механизм для хвостовых сценариев**, не стандартная практика для каждого запуска.

Применяйте его, только если:
- запускаете 3+ агентов параллельно на длинные задачи (>10 минут),
- цена пропущенного зависания высока (потеря работы, блокировка pipeline),
- встроенные средства мониторинга не покрывают ваш сценарий.

Наблюдение из практики: на типовых пакетах параллельных агентов watchdog может не сработать ни разу — уведомления приходят до первой проверки. Ценность watchdog не в том, что он срабатывает каждый раз, а в том, что он **готов**, когда понадобится.

---

## Контрольные вопросы перед применением

Прежде чем разворачивать watchdog — пройдите чеклист. Если хотя бы один ответ «нет», упростите сценарий или используйте встроенные средства (`TaskGet`, `TaskList`, `Monitor`) вместо ручного watchdog.

1. Агентов ≥3? Иначе overhead больше пользы.
2. Ожидаемое время ≥10 минут?
3. Каждый агент пишет в уникальный файл (не в общую БД, не в общий коммит)?
4. Есть брифы агентов в отдельных файлах (для restart)?
5. Ни один агент не делает non-idempotent операции (commits, внешние API, БД, сообщения)?
6. Проверили, не хватит ли `TaskGet` + `ScheduleWakeup` в цикле?
7. Запас основного контекста ≥60K токенов?

Если все семь «да» — разворачиваем watchdog. Если хотя бы один «нет» — не применяйте эту методику, она не даст выигрыша.

## Проблема, которую решает

### Без watchdog

Оркестратор запускает N агентов в фоне и ждёт уведомлений. Если одно из них не пришло (сетевой сбой, зацикленный промпт, deadlock внутри агента) — оркестратор ждёт бесконечно. Когда ожидание явно затянулось, начинаются слепые действия: перезапускать всех? Одного? Какого?

### С watchdog

Оркестратор сохраняет состояние запуска в файл на диске, планирует периодическую проверку через `ScheduleWakeup`, и на каждой проверке смотрит: обновляется ли файл output у агента? Если файл не менялся дольше порога — агент считается зависшим и перезапускается по сохранённому промпту. Старый агент не убивается: если он вернётся позже, результат разрешается по времени модификации.

**Объективный критерий живучести:** файл output агента перестал обновляться дольше `stale_threshold`.

---

## Встроенные альтернативы — проверьте сначала

Перед тем как разворачивать watchdog, проверьте, не хватит ли встроенных средств:

| Инструмент | Для чего | Когда достаточно |
|---|---|---|
| `TaskGet` / `TaskList` | Получить статус фоновых задач | Нужен опрос состояния, не event-driven реакция |
| `TaskOutput` | Прочитать накопленный output агента | Нужен текущий прогресс, не факт зависания |
| `Monitor` | Event-driven отслеживание изменений файла | Агенты пишут в предсказуемые файлы, есть паттерны прогресса |
| Базовые уведомления `Agent` | Прийти только при завершении | Задача короткая, риск зависания низкий |

**Правило:** если `TaskGet` в цикле с `ScheduleWakeup` закрывает вашу задачу — не пишите watchdog на stat-файлов. Ручной watchdog нужен, когда встроенных инструментов не хватает (например, задача очень длинная и важна быстрая реакция на stale output, а не только на факт завершения).

---

## Когда НЕ применять

- **1-2 агента** — проще дождаться уведомлений, накладные расходы больше пользы.
- **Короткие задачи (<5 минут)** — watchdog overhead больше выигрыша.
- **Non-idempotent агенты** — если агент делает commits, вызывает внешние API, пишет в БД или отправляет сообщения, restart создаёт дубли и побочные эффекты. Для таких агентов restart запрещён в принципе.
- **Общее состояние** — два агента пишут в один файл, одну таблицу, один репозиторий. Restart создаёт race conditions.
- **Stateful pipeline** — агенты зависят друг от друга последовательно, restart одного ломает цепочку.
- **Оператор активно мониторит** — ручной restart быстрее.

---

## Архитектура

Три компонента:

1. **`Agent` с `run_in_background: true`** — запускает агента в изолированном контексте (main не тратится), возвращает идентификатор и путь к output-файлу. Уведомление о завершении приходит отдельно.
2. **`ScheduleWakeup`** — планирует следующую проверку через N секунд (лимит 60–3600). Работает как heartbeat fallback: даже если уведомления не приходят, оркестратор проснётся и проверит сам.
3. **State file на диске** — persistent memory между проверками. Хранит список запусков, ожидаемые дедлайны, метрики последней проверки. Критично для случая, когда сессия компактизируется: новая сессия читает state file и продолжает watchdog с того же места.

### Поток данных

```
Старт:
  Для каждого агента:
    Agent(run_in_background=true) → agentId + output path
    Запись в state file
  ScheduleWakeup через 20 минут

Проверка:
  Загрузить state file
  Для каждого агента со статусом "running":
    stat(output_file) → mtime, size
    Сравнить с last_check_mtime
    Если stale → пометить на restart
    Обновить метрики
  Сохранить state file
  ScheduleWakeup через 10 минут (или остановка, если все завершились)
```

## Пошаговый pipeline

### Шаг 1. Запуск агентов

Один message — несколько параллельных вызовов `Agent` с `run_in_background: true`. В брифе каждому агенту обязательно:

- Явный output path — куда писать финальные файлы.
- Пометка «это критично, сохраняй только сюда».
- Time budget («~25 минут»).
- Ожидаемый формат результата.
- Запрет трогать общее состояние и делать non-idempotent операции (если вы закладываете возможность restart).

### Шаг 2. Сохранение state

Сразу после возврата `Agent` запишите состояние в JSON:

```json
{
  "started_at": 1776177811,
  "expected_total_minutes": 35,
  "check_interval_seconds": 600,
  "stale_threshold_seconds": 600,
  "elapsed_threshold_seconds": 1200,
  "max_restarts_per_agent": 2,
  "agents": [
    {
      "id": "<agentId>",
      "name": "<role>",
      "subagent_type": "<type>",
      "status": "running",
      "prompt_key": "<key>",
      "expected_deadline_at": 1776180000,
      "output_file": "<output-dir>/<agentId>.output",
      "expected_result_files": ["<project-tmp>/result-1.md"],
      "last_check_at": 0,
      "last_check_mtime": 0,
      "last_check_size": 0,
      "restart_count": 0,
      "final_result": null
    }
  ]
}
```

Брифы агентов храните отдельными файлами в `<project-tmp>/agent-prompts/<key>.md`, чтобы restart мог их переиспользовать.

### Шаг 3. Первая проверка

Первый `ScheduleWakeup` — на 20 минут (агентам нужно время на warmup; раньше — много false positives).

```
ScheduleWakeup(
  delaySeconds=1200,
  reason="Watchdog check #1",
  prompt="/loop watchdog check"
)
```

### Шаг 4. Проверка на wakeup

Логика проверки для каждого агента со статусом `running`:

1. **Есть ли output-файл?** Если нет и прошло больше 5 минут от старта — пометить `needs_restart`.
2. **Когда файл менялся в последний раз?** Считаем `stale_for = now - mtime(output_file)`.
3. **Два условия stale одновременно:** `stale_for > stale_threshold` (10 минут по умолчанию) **и** общее время работы `elapsed > elapsed_threshold` (20 минут). Оба условия нужны, чтобы не считать stale агентов, которые ещё в warmup.
4. **Если оба условия — проверить хвост файла** на маркер «завершено»: `"status": "completed"`, `completed successfully` или аналогичный — ваш агент должен писать такой маркер в последних строках при успешном завершении. Если маркер есть — ставим статус `completed_silently` (завершился, но уведомление не пришло). Если маркера нет — `needs_restart`.
5. **Hard timeout.** Если общее время работы превысило ожидаемый дедлайн плюс буфер (900 секунд) — `needs_restart`, даже если файл ещё обновляется.
6. **Обновить метрики в state file:** `last_check_mtime`, `last_check_size`.

Реализация — любой удобный язык (Python, bash + `jq`, Node). Референс-реализацию на Python (~40 строк) можно запросить у Claude Code фразой «напиши watchdog-check по логике рецепта `parallel-agents-watchdog.md`, шаг 4, Python». Рецепт описывает протокол, а не конкретный листинг — выбор языка за вами.

**Важное предупреждение про маркер «завершено».** Протокол рассчитывает, что агенты пишут заранее согласованную строку в хвост output-файла при успешном завершении. Если агент этого не делает — рецепт не сработает, нужно либо инструктировать агентов писать маркер явно в брифе, либо использовать вместо watchdog встроенные `TaskGet`/`TaskList`, которые знают о статусе агентов без магических строк.

### Шаг 5. Restart stale агентов

Для каждого агента с `needs_restart`:

1. Прочитать брифинг из `<project-tmp>/agent-prompts/<key>.md`.
2. Добавить префикс: `RESTART #N. Предыдущий запуск завис. Проверь, не успел ли он записать результат в <output>. Если успел — заверши работу с "already done". Если нет — начни с нуля.`
3. Вызвать `Agent` заново с тем же `subagent_type`.
4. Обновить state: новый agentId, `restart_count += 1`.

**Важно:** старого агента не убивайте. Если он вернётся позже — разрешите конфликт по mtime в safe apply (шаг 6). Если `restart_count` превысил `max_restarts_per_agent`, помечайте `failed` и выходите: дальше нужно вмешательство оператора.

### Шаг 6. Safe apply

Когда результаты применяются в production, для каждого ожидаемого output-файла:

```python
import glob, os, shutil
from pathlib import Path

candidates = glob.glob(f"{tmp_dir}/*.result.md")
latest = max(candidates, key=os.path.getmtime)
shutil.copy(latest, production_path)
```

Самая свежая версия по mtime — побеждает. Это корректно разрешает ситуацию, когда старый агент вернулся после restart.

### Шаг 7. Следующая проверка или остановка

Если хоть один агент ещё `running` — планируйте следующий `ScheduleWakeup` через 10 минут. Если все завершились — **не планируйте новый wakeup**, loop останавливается.

---

## Пороги по умолчанию

| Параметр | Значение | Почему |
|---|---:|---|
| Первая проверка | 1200 с (20 мин) | Warmup агентов, меньше false positives |
| Интервал проверок | 600 с (10 мин) | Баланс реакции и overhead |
| `stale_threshold` | 600 с (10 мин) | Файл не меняется 10 мин — подозрительно |
| `elapsed_threshold` | 1200 с (20 мин) | Комбинируется со stale — второе условие |
| `max_restarts_per_agent` | 2 | Дальше — вмешательство оператора |
| Hard timeout buffer | +900 с к дедлайну | Шанс медленному агенту |

Для задач длиннее 35 минут увеличивайте пороги пропорционально: `timeout ≈ max_expected × 1.5`.

---

## Edge cases

**1. Уведомление пришло после restart.**
Старый агент завершил работу после того, как был запущен новый. Решение: на wakeup проверяйте оба идентификатора; результат разрешается по mtime (шаг 6).

**2. Output-файл не создан вообще.**
Агент упал сразу. Решение: если `elapsed > 5 минут` и файла нет — считайте stale, restart.

**3. Агент пишет, но очень медленно.**
Крупный аналитический агент пишет отчёт медленно, mtime обновляется раз в 2–3 минуты. Порог 10 минут это покрывает. Если агент пишет менее 1 байта в минуту — это действительно зависание.

**4. Сессия компактизировалась.**
Main context потерял state, но файлы на диске остались. Решение: state file живёт **на диске**, не в контексте. Новая сессия читает его и продолжает watchdog.

**5. Все агенты одновременно помечены stale.**
Вероятно, network flakiness или общий сбой. Restart всех сразу нагрузит систему. Решение: порог «≥3 stale одновременно» → пауза 5 минут, потом restart только самых старых.

**6. Non-idempotent агент завис.**
Такого быть не должно: вы не закладывали возможность restart для таких агентов (см. «Когда НЕ применять»). Если всё же случилось — не restart, а ручной разбор. Пометьте `failed`, уведомите оператора.

## Интеграция в сеансовую эстафету

Если в проекте используется [`session-handoff.md`](session-handoff.md) (сеансовая эстафета), после завершения watchdog-pipeline зафиксируйте результат в **§3 «Что закрыто»** эстафетного файла. Если задача watchdog-сборки эстафеты — **финальный синтез делает основной поток сессии, не subagent**: субагент видит только свои отчёты, теряет тон оператора и реальные решения. Это эмпирическое правило подтверждено в проектах с многоагентной инвентаризацией.

Минимальный блок для §3:

```markdown
### Watchdog-запуск
- Агентов запущено: N
- Проверок watchdog: M
- Stale detected: K
- Restarts: L
- Overhead основного контекста: ~<tokens>K
- Результат: <все завершились / N failures после max_restarts>
```

Это даёт проектную память для калибровки порогов в будущих запусках.

## Применимость в разных сферах

- **Разработка** — batch-аудит кода, массовое обновление конфигов, параллельный рефакторинг.
- **Аналитика** — параллельный разбор N датасетов или отчётов, каждый агент → свой файл.
- **Юриспруденция** — параллельный review десятков документов (требует помощи разработчика для развёртывания).
- **Операции и менеджмент** — параллельный аудит регламентов, процессов, SOP-документов.

**Не подходит:**

- **Контент** — генерация обычно короткая (<10 минут), watchdog избыточен.
- **Любая работа с общим состоянием** (одна БД, одна ветка git) — restart создаёт конфликты.

## Appendix: шаблоны брифов

Готовые брифы для трёх типовых ролей агента. Копируйте и заполняйте плейсхолдеры под свою задачу.

### A. Отчётный агент (анализ, audit)

```
Ты — <agent_type>. Задача: <описание>.

Output file (сохраняй ТОЛЬКО сюда):
<full_output_path>

Если файл уже существует (другой агент написал раньше) — верни
"skipped, already exists" и завершись.

Time budget: ~<N> минут.

Инструменты: Read, Grep, Glob, Bash, Write.

Не делать: <запреты, например "только анализ, не менять код">.

Структура отчёта:
1. Резюме (3-5 строк)
2. Найденные пункты
3. Рекомендации
4. Путь к сохранённому файлу в финальном ответе

В конце работы — запиши в последней строке output файла маркер:
"status": "completed". Он нужен watchdog-протоколу для определения
успешного завершения без уведомления.
```

### B. Writer-агент (генерация файла)

```
Ты — <agent_type>. Задача: написать <тип файла>.

Output file: <full_output_path>

Обязательно прочитай перед написанием:
1. <existing file>
2. <context file>

Требования: <перечень>. Объём: <диапазон> строк.

Не делать:
- Не редактировать существующие файлы напрямую
- Не тратить токены на reinvestigation — читай контекст первым делом

Time budget: ~<N> минут.

В конце: запиши маркер "status": "completed" в последней строке файла.
```

### C. Backup-агент (защита от failure основного)

```
Ты — <agent_type>. Параллельно работает другой агент, который пишет
<target file>.

Правило: перед началом работы проверь, существует ли <target file>.
Если да — верни "already written" и завершись. НЕ дублируй работу.

Если основной агент завис (файла нет через N минут) — пиши сам с тем же
брифом:

<основной бриф>

Time budget: 5 минут на проверку + <N> минут на работу, если пишешь сам.

В конце: маркер "status": "completed" в последней строке.
```

**Маркер завершения критичен для watchdog-протокола:** без него шаг 4 проверки не сможет отличить «завершился, но уведомление потерялось» от «завис». Добавляйте его во все брифы.
