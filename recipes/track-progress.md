# Отслеживание прогресса

## Принцип

Отслеживание прогресса — это **автоматический сервис Claude Code**, а не задача для пользователя. Как автосохранение в текстовом редакторе: работает само, пользователь о нём не думает.

## Что делает Claude Code (автоматически, без участия пользователя)

При завершении сессии или при паузе — Claude Code записывает в CLAUDE.md одну строку:

```markdown
<!-- [claude-code-practices v1.0] BEGIN: current-step -->
**Последний шаг:** [краткое описание того, над чем работали]
<!-- [claude-code-practices v1.0] END: current-step -->
```

При следующем запуске — Claude Code читает эту строку и продолжает с того же места. Пользователю ничего делать не нужно.

## Строгие правила

1. **НЕ создавай файлы трекинга** (current-sprint.md, project-plan.md, global-plan.md) — если пользователь сам об этом не попросил
2. **НЕ создавай фазы, этапы, подзадачи** — если пользователь не описал свой проект как многоэтапный
3. **НЕ выдумывай бизнес-процессы пользователя** — никаких «при получении брифа», «согласование с клиентом», «фаза оптимизации»
4. **НЕ предлагай шаблоны для заполнения** — пользователь не должен заполнять никакие файлы вручную
5. Одна строка в CLAUDE.md — достаточно для 90% задач

## Когда предлагать расширенный трекинг

**Только если пользователь сам говорит** что-то вроде:
- «У меня проект на несколько недель»
- «Как не потерять нить в большом проекте?»
- «Мне нужен план работ»

В этом случае — **спроси**, что за проект, какие этапы он видит сам. Не придумывай за него. Потом предложи:

> Могу создать файл с планом вашего проекта — этапы, задачи, статусы. Claude Code будет сам обновлять прогресс. Хотите?

И только после подтверждения — создай план на основе того, что рассказал пользователь.

## Для продвинутых — только для разработчиков, только по запросу

Если пользователь ведёт сложный многоэтапный проект и хочет детальный трекинг:

- **Указатель в CLAUDE.md** — одна строка «что делаем сейчас» (создаётся автоматически)
- **Оперативный трекер** — файл с подзадачами текущего этапа (создаётся по запросу)
- **Генеральный план** — файл со всеми этапами и общим прогрессом (создаётся по запросу)

Все три файла обновляются Claude Code автоматически — пользователь не редактирует их вручную.

### Маркер позиции в оперативном трекере

В оперативном трекере отмечайте текущий шаг маркером `← ТУТ`:

```markdown
- [x] Настроить базу данных
- [x] Создать API-эндпоинты
- [ ] Написать тесты для API ← ТУТ
- [ ] Интеграция с фронтендом
```

Claude Code при старте сессии находит маркер и сразу понимает, где продолжать. При завершении шага — маркер автоматически переносится на следующий.

### Hook на старте сессии — гарантированная загрузка контекста

> Hook — это автоматическое действие, которое выполняется при старте сессии. Если вы не работаете с кодом, эта секция вам не нужна — базового отслеживания (одна строка в CLAUDE.md) достаточно.

Инструкции в CLAUDE.md зависят от модели — она может их пропустить. Hook выполняется средой исполнения и работает со стабильностью ~99%. Для гарантированной загрузки контекста при старте сессии добавьте в `.claude/settings.json`:

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "cat _docs/project-brief.md"
      }]
    }]
  }
}
```

Результат команды попадает в контекст Claude Code автоматически. Подставьте свой файл вместо `_docs/project-brief.md` — это может быть план, брифинг или оперативный трекер.

### Обязательная проверка dev_log при старте сессии

> Этот паттерн применим, только если вы явно решили вести журнал разработки. По умолчанию Claude Code НЕ создаёт dev_log — используется одна строка-указатель в CLAUDE.md (описано выше).

Паттерн для проектов, где ведётся журнал разработки. Добавьте в CLAUDE.md блок, который заставляет Claude Code проверять непрерывность истории при каждом запуске:

```markdown
### Обязательная проверка dev_log
При начале новой сессии — проверь, записана ли история предыдущей сессии (дня)
в `_docs/dev_logs/`. Если нет session-summary за предыдущий рабочий день —
создай его ПРЕЖДЕ чем начинать новую работу.
Также обнови `project-chronicle.md` и лог сессий в `current-sprint.md`.
```

**Зачем:** без этого блока контекст предыдущих сессий теряется. Claude Code начинает работу «с чистого листа» и может повторить уже решённые задачи или противоречить ранее принятым решениям.

**Как работает:** при каждом старте Claude Code читает CLAUDE.md, видит инструкцию, проверяет наличие логов. Если лог отсутствует — восстанавливает контекст из git history, файлов проекта и памяти, записывает summary, и только потом переходит к новой задаче.

**Адаптация под ваш проект:** замените пути и имена файлов на свои. Важен принцип, а не конкретная структура:
- `_docs/dev_logs/` → папка, где хранятся логи сессий (может быть `.ai-factory/logs/`, `docs/sessions/` и т.д.)
- `project-chronicle.md` → общая хроника проекта (может быть `CHANGELOG.md`, `docs/history.md`)
- `current-sprint.md` → оперативный трекер (может быть `TODO.md`, `PLAN.md`)

**Ограничение:** как и любая инструкция в CLAUDE.md, это не гарантия выполнения на 100%. В длинных сессиях или на слабых моделях инструкция может быть пропущена. Если непрерывность логов критична — дублируйте проверку в хуке (hook) на старте сессии.

### Синхронизация контекста перед очисткой сессии

Сессии Claude Code имеют ограниченный контекст. Когда он заполняется — нужна очистка (`/clear`) или новая сессия. Но в контексте накопилась информация, которая нигде не записана: принятые решения, обнаруженные проблемы, промежуточные выводы. Если просто очистить — всё это потеряется.

**Промпт перед очисткой:**

```
Проведи повторный анализ всей структуры файлов, особенно тех, которые
создавались или менялись в текущей сессии. Проверь, что вся важная информация,
влияющая на состояние проекта, имеет своё отражение в многоуровневой памяти
проекта (CLAUDE.md → memory → файлы проекта).

Убедись, что ничего не пропущено и вся информация надёжно записана на диск.
Это ВАЖНО, потому что мы собираемся сделать очистку сессии.
```

**Что делает Claude Code по этому промпту:**
1. Сканирует файлы, которые менялись в сессии
2. Сверяет с тем, что записано в CLAUDE.md и памяти
3. Дописывает недостающее: решения, статусы, выводы
4. Обновляет указатель «последний шаг» в CLAUDE.md
5. После подтверждения — можно безопасно делать `/clear`

**Когда использовать:**
- Сессия стала медленной (контекст заполнен)
- Переключаетесь на другую задачу в том же проекте
- Заканчиваете рабочий день
- Перед `git commit` крупных изменений — чтобы зафиксировать не только код, но и контекст решений

**Многоуровневая память — куда что пишется:**

| Уровень | Что хранит | Пример |
|---------|-----------|--------|
| CLAUDE.md | Текущий статус, последний шаг | `Последний шаг: рефакторинг auth-модуля` |
| Memory (`.claude/memory/`) | Решения, факты, предпочтения | `Выбрали JWT вместо сессий, потому что...` |
| Файлы проекта | Dev-логи, хроники, планы | `_docs/dev_logs/2026-03-21.md` |

Ключевой принцип: **после синхронизации новая сессия должна восстановить полный контекст, не задавая вопросов.** Если Claude Code в новой сессии спрашивает «чем мы занимались?» — синхронизация была неполной.
