# Как создавать кастомных агентов

## Что такое агент

Агент — md-файл с описанием роли, экспертизы и правил поведения. Claude Code вызывает его через Agent tool, и AI работает в рамках заданной роли, а не как универсальный генералист.

## Где хранить агентов

Агентов можно хранить в двух местах:

**Глобально** — `~/.claude/agents/` — агент доступен во всех проектах.

**В проекте** — `.claude/agents/` (в корне проекта) — агент доступен только в этом проекте.

**Какой вариант выбрать:**
- Универсальные роли (ревьюер, редактор) → глобально
- Проектные роли (знающие конкретный стек/данные) → в проекте

**Префиксы для проектных агентов:** если в проекте несколько агентов — используйте префикс с именем проекта: `proj-code-reviewer.md` вместо `code-reviewer.md`. Это предотвращает конфликты имён с глобальными агентами и агентами из других проектов. Формат: `[проект]-[роль].md`.

**Важно для Windows:** на некоторых конфигурациях Windows запись в глобальную папку `~/.claude/agents/` может быть заблокирована (ошибка `EPERM: operation not permitted`). Если это произошло:
1. **Попробуй проектный вариант** — создай `.claude/agents/` в корне проекта. Права на запись в проект обычно есть
2. Если нужен именно глобальный агент — попроси пользователя создать файл вручную (через Блокнот, VS Code или Проводник) и покажи содержимое для копирования

## Когда создавать агента

- Задача повторяется регулярно (анализ, проверка, оформление)
- Нужна конкретная экспертиза (бухгалтер, юрист, редактор, аналитик)
- Важно, чтобы AI следовал определённому протоколу (чеклист, порядок действий, формат вывода)
- Над задачей работают несколько ролей, и вы хотите "коллегиальную" работу

Не нужен агент для разовой задачи — достаточно промпта.

## Проверка перед созданием

**Перед созданием любого агента:**

1. Определи, куда записывать: глобально (`~/.claude/agents/`) или в проект (`.claude/agents/`). Спроси пользователя, если неясно
2. Проверь, есть ли файл с таким именем в выбранной папке
3. Если файл существует — покажи текущее содержимое и спроси: обновить (с бэкапом .bak), переименовать новый, или оставить как есть
4. Если файла нет — создай

**Если запись не удалась** (ошибка `EPERM`, `Access Denied`, `operation not permitted`):
- Попробуй проектный вариант `.claude/agents/` — туда права обычно есть
- Если и это не помогло — покажи пользователю готовое содержимое файла и попроси создать вручную: «Создайте файл [путь] с таким содержимым [текст]. Можно через Блокнот, VS Code или любой текстовый редактор. Когда создадите — скажите "готово".»

## Структура файла агента

```markdown
---
name: имя-агента
description: Одна строка — когда вызывать этого агента
model: opus | sonnet | haiku  # опционально, для контроля бюджета
source: claude-code-practices v1.0  # маркер принадлежности к пакету
---

# Роль
Ты — [роль]. Твоя экспертиза: [области].

# Контекст
[Описание предметной области, терминология, источники данных]

# Правила
1. [Правило поведения]
2. [Правило поведения]
...

# Формат вывода
[Как должен выглядеть результат работы агента]

# Примеры триггеров
- "[фраза пользователя]" → этот агент
```

## Примеры агентов для разных областей

### Финансовый аналитик
**Роль:** анализ P&L, cashflow, налоговых рисков.
**Ключевые правила:**
- Каждая цифра подтверждена расчётом или ссылкой на источник
- Формат вывода: таблицы с итогами, отклонения выделены
- Разделять факты и интерпретации
- При обнаружении аномалий — сигнализировать, а не молча включать в отчёт
**Модель:** opus (нужен глубокий анализ)

### Редактор / корректор
**Роль:** вычитка текстов, стилистика, структура.
**Ключевые правила:**
- Сохранять авторский стиль, не переписывать
- Замечания с обоснованием (не "плохо", а "двусмысленная формулировка, потому что...")
- Разделять критические ошибки (факты, логика) и стилистические
**Модель:** sonnet (процедурная задача)

### Аудитор / ревьюер
**Роль:** проверка документов, отчётов, данных по чеклисту.
**Ключевые правила:**
- Работает по заданному чеклисту, не пропускает пункты
- Выдаёт протокол: что проверено, что найдено, рекомендации
- Разделяет блокеры (нельзя выпускать) и замечания (желательно исправить)
**Модель:** opus (нужна внимательность к деталям)

### Исследователь / аналитик данных
**Роль:** поиск информации, систематизация, подготовка справок.
**Ключевые правила:**
- Указывать источники и степень достоверности
- Структурировать находки по категориям
- Отмечать пробелы в данных ("не удалось найти X")
**Модель:** sonnet

### Менеджер проектов / координатор
**Роль:** отслеживание статуса задач, выявление рисков, подготовка статусных отчётов.
**Ключевые правила:**
- Отслеживать зависимости между задачами — если задача A блокирует B, сигнализировать
- Статус-отчёты: факты и метрики, не ощущения («выполнено 7 из 12» вместо «почти готово»)
- При обнаружении отклонения от плана — сразу предложить варианты (перераспределить, сдвинуть срок, сократить скоуп)
**Модель:** sonnet (процедурная координация)

## Граница ролей — кто создаёт, тот не оценивает

Агент, который создаёт результат, не должен сам его оценивать. Разработчик не ревьюит свой код. Ревьюер не пишет код. Аналитик не разрабатывает.

Это тот же принцип, что и в аудите: независимость оценки. Если один агент и генерирует, и проверяет — он найдёт ровно те ошибки, которые "не мешают". Слепые зоны автора — слепые зоны проверки.

**Как прописать в файле агента:**

```markdown
# Границы роли
- Ты НЕ пишешь код — ты его проверяешь
- Если нужно внести правку — опиши, что изменить, но не редактируй сам
- При конфликте интересов (создал и проверяешь) — сообщи об этом
```

**Правило:** если один агент создал артефакт — для проверки вызывайте другого. Даже если это "лишний" вызов — он дешевле пропущенной ошибки.

## Гигиена агентов — четыре правила против протухания

Описание агента — «стабильный» документ, и любой изменчивый факт внутри него протухает (общий принцип — [`fact-ownership.md`](fact-ownership.md)). Четыре правила, выведенные из аудита зрелого проекта:

**1. Общие правила — ссылкой, не копией.** Один и тот же блок (токен-оптимизация, протокол отчёта), скопированный в несколько агентов, расходится: в одном правят, в остальных — нет; появляются опечатки и рассинхрон с CLAUDE.md. Правило живёт в одном месте; агент — однострочная ссылка: «правила вывода — см. CLAUDE.md, секция X». Осознанное исключение допустимо (например, критичное правило автономии дублируется намеренно) — но тогда зафиксируйте это решение письменно.

**2. Факты среды не хардкодить.** Адреса, порты, версии систем, ставки и нормативы — ссылкой на reference-файл памяти: «все реквизиты — в reference_X, сюда не копировать». Хардкод протухает при первой переустановке среды или смене регламента. В регуляторных доменах это опаснее всего: на проекте бухгалтерской аналитики налоговая ставка, захардкоженная в трёх агентах, устарела при смене регионального закона — агенты продолжали уверенно советовать по старой ставке.

**3. Уроки решённых задач — ссылками на память.** Секция «Опыт» с зашитыми кейсами мертвеет: память проекта пополняется, а тело агента — нет. Паттерн: «уроки — читать memory-файлы по триггеру задачи: [список ссылок]»; новые уроки добавляются в память по чек-листу закрытия задачи, не в тело агента.

**4. Секции «Когда использовать / Когда НЕ использовать»** — в каждом агенте, с маршрутизацией на соседних: «для задач X вызывай агента Y». Без них пользователь зовёт не того специалиста, а агенты дублируют друг друга.

## Коллегиальная работа — несколько агентов на одну задачу

Один AI не может быть одновременно бухгалтером, финдиректором и аудитором. Можно запустить несколько агентов с разными ролями:

**Пример — финансовая аналитика (3 волны):**
1. **Параллельно:** агент-бухгалтер (сводка операций), агент-аналитик (P&L), агент-налоговик (риски)
2. **Синтез:** сопоставление результатов, разрешение противоречий
3. **Оформление:** итоговый отчёт

**Пример — подготовка документа:**
1. **Параллельно:** агент-исследователь (сбор фактов), агент-аналитик (выводы)
2. **Синтез:** агент-редактор (сборка, стилистика, вычитка)

Ключевое: каждый агент видит только свою роль и даёт экспертизу в ней.

## Тиринг моделей

| Модель | Для чего |
|--------|----------|
| opus | Сложный анализ, принятие решений, работа с нюансами |
| sonnet | Процедурные задачи, оформление, работа по шаблону |
| haiku | Простые проверки, извлечение данных, мониторинг |

В сессии с множеством субагентов тиринг значительно снижает расход — зависит от соотношения задач и моделей.

## Профессиональная автономия

Агенты — не слепые исполнители. Пропишите в правилах каждого агента:

> «Три режима реакции на задачу:
> 1. Есть лучше — предложи альтернативу с обоснованием, выполни после выбора
> 2. Нецелесообразно — аргументируй отказ (минимум 3 аргумента), предложи что делать вместо
> 3. Согласен — подтверди, дополни деталями, выполни
> Слепое согласие — антипаттерн.»

Подробнее о принципах поведения: `recipes/behavior-rules.md`

## Как создавать агентов на лету

Не нужно проектировать заранее. Работаете в сессии, накопили контекст — говорите Claude Code:

> "Создай нового агента `[имя]` на основе нашего текущего контекста. Роль: [роль]. Сохрани в `~/.claude/agents/[имя].md`"

Агент создаётся из накопленного опыта, без перезапуска.

## Агенты-персоны пользователей

Отдельный тип агентов — персоны ваших пользователей. Они не анализируют UX со стороны, а говорят от лица конкретного человека: покупателя, менеджера, оператора. Полезны для отладки бизнес-идей, проверки интерфейсов, валидации гипотез.

Подробнее: `recipes/user-personas-as-agents.md`
