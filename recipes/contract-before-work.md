# Контракт до реализации — определи результат заранее

> Этот рецепт — для всех сфер. Для разработчиков есть технический аналог: `recipes/tdd-for-developers.md`.

## Идея

Прежде чем делать работу — определите, как выглядит правильный результат. Не «сделать отчёт», а «отчёт считается готовым, когда выполнены эти 7 критериев». Это звучит очевидно, но на практике большинство задач начинаются без чётких критериев. Результат: переделки, споры «а я думал, что...», потерянное время.

В разработке этот подход называется TDD (Test-Driven Development) — тесты пишутся до кода. В других сферах принцип тот же: **критерии успеха определяются до начала работы**.

## Как это работает

### Шаг 1: Сформулировать критерии готовности

Прежде чем начать работу, ответьте на вопрос: «Как я пойму, что результат правильный?»

Не абстрактно («отчёт должен быть хорошим»), а конкретно:
- Какие данные должны быть включены?
- Какой формат вывода?
- Какие проверки должен пройти результат?
- Что является ошибкой?

### Шаг 2: Оформить как чеклист

Критерии оформляются как чеклист — список пунктов, каждый из которых можно проверить однозначно (да/нет).

### Шаг 3: Выполнить работу

Теперь Claude Code работает с чётким ориентиром. Он знает, что результат должен пройти все пункты чеклиста.

### Шаг 4: Проверить по чеклисту

После завершения работы — пройти по чеклисту. Каждый пункт — проверен. Если что-то не пройдено — доработать.

## Примеры для разных сфер

### Финансовая аналитика

**Задача:** квартальный отчёт P&L.

**Контракт (критерии готовности):**
- [ ] Все операции за период включены (сверка с банковской выпиской)
- [ ] Каждая итоговая цифра подтверждена расчётом или ссылкой на источник
- [ ] Валютные операции пересчитаны по курсу ЦБ на дату операции
- [ ] Категоризация расходов соответствует утверждённому справочнику
- [ ] Отклонения >10% от предыдущего квартала — объяснены
- [ ] Формат: таблицы с разделителем тысяч, 2 знака после запятой
- [ ] Итоги по разделам сходятся с общим итогом

### Исследование / аналитическая справка

**Задача:** обзор рынка облачных сервисов.

**Контракт:**
- [ ] Охвачены топ-5 провайдеров по доле рынка
- [ ] Для каждого: цены, SLA, регионы, сертификации
- [ ] Все данные — из официальных источников (ссылки обязательны)
- [ ] Дата актуальности каждого источника указана
- [ ] Сравнительная таблица по единым критериям
- [ ] Рекомендация с обоснованием (не «лучший», а «лучший для [наших задач], потому что...»)

### Редактура / подготовка текста

**Задача:** вычитка статьи перед публикацией.

**Контракт:**
- [ ] Фактические ошибки выявлены и исправлены (даты, цифры, имена)
- [ ] Орфография и грамматика проверены
- [ ] Стилистика единообразна (не смешивать «вы» и «ты», формальный и разговорный стиль)
- [ ] Все аббревиатуры расшифрованы при первом упоминании
- [ ] Логическая структура: тезис → аргументы → вывод в каждом разделе
- [ ] Длина абзацев — не более 5-7 строк (читаемость)

### Юридический анализ

**Задача:** проверка договора перед подписанием.

**Контракт:**
- [ ] Все существенные условия присутствуют (предмет, цена, сроки, ответственность)
- [ ] Штрафные санкции симметричны (обязательства обеих сторон)
- [ ] Условия расторжения — явные, без «ловушек»
- [ ] Юрисдикция и порядок разрешения споров — определены
- [ ] Персональные данные — обработка соответствует 152-ФЗ
- [ ] Сроки оплаты и порядок приёмки — однозначные

### Медицинский анализ данных

**Задача:** анализ результатов клинического исследования.

**Контракт:**
- [ ] Выборка описана: размер, критерии включения/исключения
- [ ] Статистические методы соответствуют типу данных
- [ ] p-value и доверительные интервалы указаны для каждого вывода
- [ ] Ограничения исследования перечислены явно
- [ ] Конфликт интересов — заявлен или отмечено отсутствие

## Как попросить Claude Code

> Прежде чем начать работу над [задача], составь чеклист критериев готовности. Что должно быть выполнено, чтобы результат считался правильным? Потом выполни работу и проверь по этому чеклисту.

Или более подробно:

> Я хочу [задача]. Перед тем как начинать:
> 1. Сформулируй 5-10 критериев готовности (конкретных, проверяемых)
> 2. Покажи мне — я скорректирую, если нужно
> 3. Выполни работу
> 4. Проверь результат по каждому критерию
> 5. Покажи итог с отметками по чеклисту

## Зачем это нужно

- **Ясность** — вы и AI одинаково понимаете, что считается «готовым»
- **Качество** — ни один критерий не пропущен
- **Экономия времени** — меньше переделок, споров, уточнений
- **Прозрачность** — результат проверяем, а не «вроде нормально»
- **Повторяемость** — чеклист можно сохранить как навык и использовать для аналогичных задач
