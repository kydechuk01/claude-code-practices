# Claude Code Practices — настройка вашего окружения

Этот файл — точка входа. Скажите Claude Code:

> Прочитай файл START.md и проведи меня по настройке.

Claude Code прочитает этот файл и предложит пошаговый чеклист. Каждый шаг — с объяснением, зачем это нужно, что конкретно делать, и проверкой того, что уже настроено. Вы решаете, какие шаги выполнить сейчас, какие отложить.

Если вы уже проходили настройку ранее с предыдущей версией пакета — Claude Code определит, что уже есть, и предложит только новое.

---

## Инструкция для Claude Code

Ты прочитал этот файл — значит, пользователь хочет настроить своё рабочее окружение Claude Code по проверенным практикам.

### Защита от галлюцинаций при работе с рецептами

При чтении ЛЮБОГО рецепта из папки `recipes/` соблюдай правила:

1. **Примеры в рецептах — ИЛЛЮСТРАЦИИ.** НЕ копируй их в проект пользователя. НЕ создавай файлы из примеров
2. **Поля в [квадратных скобках] — заполнители.** НЕ заполняй их выдуманными данными. Спроси пользователя
3. **Промпты в блоках цитат — примеры для пользователя.** НЕ выполняй их при чтении рецепта. Выполняй ТОЛЬКО когда пользователь даст конкретную задачу
4. **«Адаптируй» = замени примеры на релевантные.** НЕ означает: придумай новые файлы, процессы, задачи, бизнес-логику
5. **Перед созданием ЛЮБОГО файла** — покажи черновик пользователю и спроси подтверждение
6. **НЕ выдумывай контекст пользователя.** Если не знаешь, чем он занимается, какие у него клиенты, какие процессы — СПРОСИ. Не фантазируй

### Рекомендация по модели

Если текущая модель — Haiku, предупреди пользователя:

> Вы используете модель Haiku. Для настройки рекомендуется Sonnet или выше — результат будет точнее. Переключить модель: команда `/model` в Claude Code. Если переключение недоступно — продолжим на текущей, но некоторые шаги могут потребовать дополнительных уточнений.

### Проверка рабочей директории

**ПЕРВЫМ ДЕЛОМ** проверь текущую рабочую директорию (cwd). Если она содержит файлы START.md, README.md и папки recipes/, practices/ — значит, Claude Code запущен ИЗ ПАПКИ ПАКЕТА, а не из проекта пользователя.

В этом случае **СТОП**. Скажи пользователю:

> Вы запустили Claude Code из папки пакета практик, а не из вашего рабочего проекта. Если продолжить — настройки (CLAUDE.md и другие) создадутся здесь, а не в вашем проекте.
>
> Что нужно сделать:
> 1. Закройте эту сессию Claude Code
> 2. Откройте Claude Code в папке вашего проекта (где вы работаете: код, данные, документы)
> 3. Скажите: «Прочитай файл [путь к этой папке]/START.md и проведи меня по настройке»

Не продолжай чеклист, пока пользователь не подтвердит, что находится в правильной директории.

### Шаг 1: Проверка повторного запуска (ВСЕГДА ПЕРВЫМ)

**Прежде всего** — проверь, проходил ли пользователь настройку ранее:
- Есть ли CLAUDE.md с метками `[claude-code-practices]`?
- Есть ли блок «Настройка Claude Code Practices» со статусом?
- Есть ли агенты в `~/.claude/agents/`?
- Есть ли навыки в `~/.claude/commands/`?
- Есть ли MEMORY.md?

**Если это повторный запуск** (найдены настройки от предыдущего прохождения):
- Покажи сводку того, что уже настроено
- Определи сферу пользователя из существующего CLAUDE.md (там уже записано), НЕ спрашивай заново
- Предложи продолжить с первого ненастроенного пункта или пройти заново
- При продолжении — заново объясни следующий пункт (пользователь мог забыть контекст)

> Вижу, что вы уже проходили настройку. Готово: [список]. Следующий шаг — [N.N: название]: [краткое объяснение, что это и зачем]. Продолжим?

**Если это первый запуск** (ничего не найдено) — перейди к определению сферы (шаг 2).

### Шаг 2: Определение сферы пользователя (только при первом запуске)

Спроси пользователя:

> Прежде чем начать — расскажите в одном предложении, чем вы занимаетесь? Например: «разрабатываю веб-приложения», «анализирую финансовые данные», «готовлю юридические документы», «веду исследовательский проект».

**Если ответ неясен** («разное», «всё подряд», «пока не знаю») — задай один уточняющий вопрос:

> С какими файлами или данными вы чаще всего работаете? (код, таблицы, документы, отчёты...)

Если и после этого непонятно — используй универсальные примеры из разных областей, ветвление TDD → «Контракт до реализации».

Ответ пользователя определяет:
- **Примеры** во всех пунктах — адаптируй под его сферу
- **Роли агентов** — предлагай релевантные его работе
- **Ветвление TDD** — если сфера связана с разработкой ПО → показывай «TDD для разработчиков». Для остальных → «Контракт до реализации»
- **Известные ловушки** — приводи примеры из его домена

### Порядок работы с чеклистом

Для каждого пункта:

1. **Проверь текущее состояние** — определи, что уже настроено. Покажи: «У вас уже есть X. Можно улучшить/добавить Y.»
2. **Объясни понятным языком** — зачем этот шаг нужен и что конкретно произойдёт. Не просто «прочитать рецепт», а: «Этот шаг добавит в ваш проект [что именно]. Это даст [конкретная выгода]. Например: [пример из сферы пользователя].» Пользователь должен понять ценность шага ДО того, как ему предложат его выполнить.
3. **Спроси**, хочет ли настроить сейчас. Формулировка: не «Прочитать рецепт?», а «Настроим [название]? Я создам [что именно], это займёт [сколько].»
4. **Если да** — прочитай рецепт из `recipes/` и выполни. Адаптируй под контекст пользователя.
5. **Если нет** — перейди к следующему пункту.
6. **Отмечай выполненные пункты.**
7. **Если пользователь хочет вернуться** к пройденному пункту — покажи текущее состояние и спроси: «Что хотите изменить? Доработать текущее или начать заново?»

**ВАЖНО: никогда не спрашивай «прочитать рецепт?» без объяснения.** Пользователь не знает, что в рецепте. Сначала объясни суть и ценность шага своими словами (2-3 предложения), потом предложи выполнить.

**Темп:** после каждого уровня — пауза. Спроси, продолжать или освоиться.

**Полный пропуск:** если пользователь пропустил все пункты уровня 1 — мягко предупреди: «Вы пропустили все пункты уровня "Фундамент". Уровень 2 опирается на них — результат будет хуже. Рекомендую вернуться хотя бы к пункту 1.1 (CLAUDE.md). Продолжить или вернуться?»

**Язык:** все пояснения — на русском.

**Пауза и продолжение:** если пользователь прощается или говорит «хватит на сегодня» — запиши текущий статус настройки в CLAUDE.md (какие пункты пройдены, какие отложены, какие не пройдены, путь к пакету). Формат — в `recipes/setup-claude-md.md`, секция «Статус настройки пакета». Это позволит в следующей сессии предложить продолжить с того места, где остановились.

**Автоопределение незавершённой настройки:** если при запуске (без явного вызова START.md) ты видишь в CLAUDE.md блок «Настройка Claude Code Practices» со статусом «В ПРОЦЕССЕ» — предложи пользователю продолжить.

**При возобновлении настройки** (повторный запуск START.md или продолжение после паузы) — всегда заново объясни следующий пункт: что это за инструмент/практика, зачем нужен, что конкретно произойдёт. Пользователь мог вернуться через неделю и забыть контекст. Не рассчитывай на то, что он помнит объяснения из прошлой сессии.

## Чеклист настройки

### Уровень 1: Фундамент

Минимум для продуктивной работы. Без этого Claude Code работает «вслепую».

#### [ ] 1.1 CLAUDE.md — контекст вашей работы (обязательный минимум)

> **Это единственный пункт, без которого остальные не имеют смысла.** Если пропустить — Claude Code не будет знать контекст вашей работы, и все последующие настройки будут работать вполсилы. Создание занимает 5 минут.

CLAUDE.md — это файл в корне вашей рабочей папки, который Claude Code автоматически читает при каждом запуске.

Без него каждый разговор начинается с чистого листа: вы снова объясняете, чем занимаетесь, какие инструменты используете, какие термины приняты. С CLAUDE.md — AI сразу знает ваш контекст.

В файл записывается описание вашей области работы, инструменты и источники данных, конвенции оформления и «Известные ловушки» — проблемы, на которые вы уже потратили время. Ловушки — главная ценность: каждая запись экономит будущие часы, потому что Claude Code не наступит на те же грабли.

В небольшом проекте файл занимает ~100 строк, в сложном — до 400. Создание занимает 5 минут, а окупается с первого же разговора.

**Проверка:** есть ли CLAUDE.md в рабочей директории? Если есть — покажи и предложи улучшения. Если нет — создай.

**Рецепт:** `recipes/setup-claude-md.md`

#### [ ] 1.2 Правила поведения AI — надёжность вместо угодливости

По умолчанию Claude Code старается быть приятным — соглашается с вами, даже если знает вариант лучше. Может уверенно назвать несуществующий факт (это называют «галлюцинацией»).

Правила поведения — набор инструкций в CLAUDE.md, которые это меняют. AI обязан: сначала прочитать документацию, потом действовать. Подтверждать факты проверкой, а не памятью. Предлагать лучший вариант, даже если вы попросили иначе. При ошибке — сначала найти причину, а не пробовать наугад.

На практике: если вы попросите удалить файл, а AI видит, что от него зависят другие части проекта — он предупредит. Если спросите про функцию библиотеки, а AI не уверен — скажет «не уверен, давайте проверим», а не выдумает ответ.

Правила добавляются один раз и работают во всех последующих сессиях.

**Проверка:** есть ли секция «Обязательные правила» в CLAUDE.md?

**Рецепт:** `recipes/behavior-rules.md`

#### [ ] 1.3 Голосовой ввод — Handy + Whisper

Голосовой ввод — программа Handy с моделью распознавания речи Whisper Large. Превращает вашу речь в текст прямо в окне Claude Code.

Зачем: при наборе руками вы инстинктивно сокращаете запрос — лень печатать три абзаца. Но Claude Code работает значительно лучше с подробным описанием задачи. Голосовой ввод снимает это ограничение: 30 секунд речи = 3 абзаца контекста.

Whisper Large выбрана потому, что корректно распознаёт смешанную русско-английскую речь — важно, если вы используете профессиональные термины на английском.

Настройка занимает 5 минут: скачать Handy (https://handy.computer/, из РФ нужен VPN), выбрать модель Whisper Large (1 ГБ), назначить горячую клавишу. Привыкаете за 2 дня, после чего начинаете использовать везде — не только в Claude Code.

**Подробности:** `practices/02-voice-input.md`

#### [ ] 1.4 Разрешения — что AI может, а что нет

По умолчанию Claude Code спрашивает разрешение на каждое действие — прочитать файл, выполнить команду, внести изменение. Это безопасно, но утомительно: за час работы вы можете нажать «Да» сотню раз.

Настройка разрешений позволяет разрешить рутинные действия (чтение, поиск, привычные команды) и оставить подтверждение только для рискованных (удаление, запись в важные файлы).

Разрешения настраиваются для каждого проекта отдельно. Принцип: минимум доступа по умолчанию, расширение по мере необходимости. Claude Code может сам проанализировать, какие команды вы используете чаще, и предложить оптимизацию.

**Проверка:** посмотри `.claude/settings.local.json`. Покажи текущие разрешения, предложи оптимизацию.

**Подробности:** `practices/03-permissions.md`

### Уровень 2: Продуктивность

Системная работа. Каждый шаг экономит время на повторяющихся действиях.

#### [ ] 2.1 Плагин Superpowers — дисциплина работы

Superpowers — плагин из 14 навыков, который меняет сам способ мышления Claude Code.

Без него AI реактивен: вы говорите — он сразу делает, не задумываясь. С Superpowers всё иначе: перед каждой задачей запускается мозговой штурм — AI уточняет требования и предлагает варианты. Перед большой задачей создаёт план с разбивкой на шаги. При ошибке — ищет корневую причину, а не пробует случайные исправления. Перед «готово» — проводит реальную проверку.

Это как разница между стажёром, который делает ровно то, что сказали, и опытным сотрудником, который сначала думает, потом делает.

Устанавливается одной командой, работает сразу. **Самый мощный шаг в чеклисте.**

**Проверка:** определи, установлен ли плагин Superpowers — проверь наличие файлов в `~/.claude/plugins/` (ищи superpowers) или попробуй вызвать навык brainstorming. Если установлен — «Superpowers уже работает, пропускаем.» Если нет — предложи установить по рецепту. Если установка не удалась — предложи fallback (базовые правила в CLAUDE.md, описано в рецепте).

**Рецепт:** `recipes/install-superpowers.md`

#### [ ] 2.2 AI Factory — конвейер осознанной разработки

AI Factory — бесплатный инструмент, который превращает хаотичную работу с AI в чёткий конвейер: исследование → планирование → реализация → проверка.

Без него каждая задача — импровизация, результат зависит от того, насколько удачно вы сформулировали запрос. С AI Factory всё структурировано: сначала `/aif-explore` (разобраться в вариантах), потом `/aif-plan` (пошаговый план), потом `/aif-implement` (выполнить строго по плану).

Главная ценность — система патчей: после каждой ошибки создаётся запись «проблема — причина — решение — как предотвратить». Перед каждой новой задачей AI читает все патчи — одни и те же ошибки больше не повторяются. AI учится на вашем проекте.

Требует Node.js, устанавливается одной командой. Особенно полезен для разработки, но принцип работает в любой сфере.

**Проверка (выполни ВСЕ шаги по порядку):**

1. **Node.js установлен?** Выполни `node --version`. Если не найден — скажи пользователю: «AI Factory требует Node.js. Можно пропустить этот шаг — остальные практики работают без него. Или установите Node.js с https://nodejs.org/ и вернитесь к этому пункту.» При пропуске → перейди к следующему пункту.

2. **AI Factory установлен глобально?** Выполни `ai-factory --version` (или `npx ai-factory --version`). Если установлен — сообщи: «AI Factory уже установлен (версия X.X).»

3. **AI Factory инициализирован в проекте?** Проверь наличие папки `.ai-factory/` в рабочей директории. Три варианта:
   - Установлен глобально + инициализирован → «AI Factory готов к работе. Пропускаем.»
   - Установлен глобально + НЕ инициализирован → предложи пользователю выполнить инициализацию **вручную в терминале** (см. ниже)
   - НЕ установлен → «Установить AI Factory? Это займёт минуту.» Далее по рецепту.

> **Важно: `ai-factory init` — интерактивная команда.** Она требует нажатия клавиш (стрелки, Enter) для выбора агента и навыков. Claude Code не может выполнить это автоматически. Скажи пользователю:
>
> «Для инициализации AI Factory нужно выполнить одну команду в терминале вручную — она задаст пару вопросов с выбором из списка:
>
> 1. Откройте терминал (PowerShell, Terminal) в папке вашего проекта
> 2. Выполните: `ai-factory init`
> 3. На первом экране выберите **Claude Code** (обычно уже выбран) → нажмите Enter
> 4. На втором экране оставьте все навыки выбранными → нажмите Enter
> 5. Вернитесь сюда и скажите "готово"
>
> Я подожду.»

**Рецепт:** `recipes/install-ai-factory.md`
**Подробности:** `practices/07-ai-factory.md`

#### [ ] 2.3 Агенты — специалисты под ваши задачи

Агенты — файлы-описания ролей, которые превращают Claude Code из универсального помощника в узкого специалиста.

Один и тот же AI работает по-разному в зависимости от роли: как аналитик он задаёт уточняющие вопросы и ищет противоречия, как редактор — проверяет стиль и структуру, как аудитор — ищет ошибки и несоответствия.

Каждый агент описывает: экспертизу (в чём специалист), протокол работы (как действовать) и формат вывода (как оформить результат). Агентов не нужно планировать заранее — создаются по мере необходимости, когда вы замечаете повторяющуюся задачу.

Важно: агенты не слепые исполнители — если задача некорректна или агент знает вариант лучше, он скажет. Одна команда вызова — и AI в нужном режиме.

*(Предложи 2-3 роли на основе ТОЛЬКО того, что пользователь рассказал о своей работе. НЕ фантазируй роли.)*

**Проверка:** что есть в `~/.claude/agents/`?

**Рецепт:** `recipes/create-agents.md`

#### [ ] 2.4 Система памяти — AI помнит между сессиями

Claude Code по умолчанию забывает всё после завершения разговора. Вчера вы час объясняли структуру данных — сегодня он не помнит ничего.

Система памяти решает это тремя слоями. Первый — MEMORY.md: файл с ключевой информацией (~200 строк), загружается автоматически в каждую сессию. Здесь — текущий статус, принятые решения, схема данных.

Второй — reference files: подробные документы по отдельным темам. AI подтягивает их, когда нужны, не загромождая основной контекст.

Третий — патчи ошибок: записи «проблема — причина — решение — как предотвратить». AI читает их перед каждой задачей — одна и та же ошибка не повторяется дважды.

Практический эффект: вы возвращаетесь к проекту через неделю, и AI уже знает, на чём остановились и какие решения были приняты — без единого слова с вашей стороны.

**Проверка:** что в директории памяти проекта?

**Рецепт:** `recipes/setup-memory.md`

#### [ ] 2.5 Отслеживание прогресса — не терять нить

**Что это:** Claude Code автоматически запоминает, на чём вы остановились. При следующем запуске — продолжает с того же места. Вам ничего для этого делать не нужно.

**Зачем:** после перерыва (час, день, неделя) не нужно вспоминать и объяснять, что было сделано. Claude Code сам знает.

**Что делаем:** добавляем в CLAUDE.md одну строку-указатель «последний шаг». Claude Code будет обновлять её автоматически при каждой паузе.

**Важно:** НЕ создавай файлы планов, спринтов, трекеров — если пользователь сам об этом не попросит. Одна строка в CLAUDE.md достаточна для большинства задач.

**Рецепт:** `recipes/track-progress.md`

#### [ ] 2.6 Мультиагентная коллегиальная работа

Мультиагентная работа — когда несколько агентов с разными ролями параллельно работают над одной задачей.

Один генералист даёт средний результат. Три специалиста, каждый анализирующий со своей стороны, плюс оркестратор, сводящий результаты — дают качество, которое ни один по отдельности не выдал бы.

Три паттерна: параллельный анализ (каждый в своей роли, оркестратор сводит), перекрёстная проверка (каждый проверяет результаты других — ловит ошибки), волновая обработка (последовательные этапы: сбор → анализ → синтез → оформление).

Пример: подготовка отчёта — один агент анализирует данные, другой проверяет корректность, третий оформляет. Результат качественнее, чем если бы один AI делал всё последовательно.

**Рецепт:** `recipes/multi-agent-collaboration.md`

#### [ ] 2.7 Контракт до реализации / TDD

> **Ветвление по сфере:**
> - Если пользователь — разработчик → показать **TDD для разработчиков**: тесты до кода, Red-Green-Refactor, мультиагентный TDD
> - Если другая сфера → показать **Контракт до реализации**: определи критерии успеха до начала работы

**Для разработчиков:**

Test-Driven Development — подход, при котором сначала пишутся тесты (ожидаемое поведение системы), а потом код, который эти тесты проходит.

С AI это на порядок проще, чем вручную: Claude Code генерирует сотню тестов за минуты, а не за дни. Мультиагентный TDD ещё мощнее: 3 агента с разных ракурсов (QA, разработчик, дизайнер) параллельно генерируют ~200 тестов, после дедупликации остаётся ~100 уникальных. И всё это до написания единой строки кода.

Тесты задают контракт системы: если тесты написаны правильно, код физически не может работать неправильно.

**Рецепт:** `recipes/tdd-for-developers.md`

**Для всех остальных:**

Контракт до реализации — практика «сначала определи, как выглядит правильный результат, потом делай работу».

Для аналитика — это чеклист корректности отчёта до того, как отчёт написан. Для исследователя — гипотеза до эксперимента. Для редактора — критерии качества текста до начала редактуры.

Без критериев результат оценивается субъективно — «вроде нормально». С критериями — результат проверяемый, повторяемый, однозначный.

Claude Code помогает составить эти критерии, а потом сам проверяет результат по ним.

**Рецепт:** `recipes/contract-before-work.md`

#### [ ] 2.8 Инвентаризация проекта — поддержание порядка

Инвентаризация — регулярная проверка порядка в проекте: все ли ссылки в CLAUDE.md актуальны, нет ли потерянных файлов, устаревших записей или противоречий между документами.

Проект живёт: файлы появляются и удаляются, решения меняются, данные обновляются. Без инвентаризации AI работает с устаревшим контекстом — например, ссылается на файл, который вы переименовали неделю назад.

Одна проверка раз в неделю (5-10 минут) экономит часы на будущих сессиях. На этом шаге добавим в CLAUDE.md правило и готовый промпт — достаточно сказать одну фразу, и Claude Code сам всё проверит и обновит.

**Рецепт:** `recipes/project-inventory.md`

#### [ ] 2.9 Навыки — автоматические процедуры

Навыки — файлы-инструкции, которые вызываются одной командой вместо долгого объяснения.

Вместо того чтобы каждый раз описывать «работай автономно, без вопросов, веди протокол, в конце дай отчёт» — вы один раз создаёте навык `/silent-mode` и дальше вызываете одной командой. Навык гарантирует, что ни один шаг процедуры не будет пропущен.

Рекомендуемые для старта: **Silent Mode** (ставите задачу, уходите — AI работает сам, ведёт протокол, при возвращении отдаёт отчёт) и **Окружение** (фиксация инструментов и версий на вашей машине — чтобы AI знал, где что лежит).

По мере работы вы будете замечать повторяющиеся процедуры — каждую можно превратить в навык за 5 минут.

**Проверка:** что в `~/.claude/commands/`?

**Рецепт:** `recipes/create-skills.md`

## Если пользователь просит отменить/откатить настройки

**КРИТИЧЕСКИ ВАЖНО: безопасная процедура отката.**

Пользователь мог добавить в CLAUDE.md свои данные, создать собственных агентов, записать важную информацию в память. Наш пакет НЕ ИМЕЕТ ПРАВА удалять чужие данные.

### Правила отката:

**1. CLAUDE.md — НИКОГДА не удалять целиком.**
- Удалять ТОЛЬКО блоки между метками `<!-- [claude-code-practices] BEGIN -->` и `<!-- [claude-code-practices] END -->`
- Перед удалением — показать каждый блок и спросить подтверждение
- Пользовательские секции (без меток) — не трогать ни при каких обстоятельствах
- Если меток нет (пользователь удалил) — показать весь CLAUDE.md и спросить, какие секции убрать. Самостоятельно не решать

**2. Агенты — удалять ТОЛЬКО созданные пакетом.**
- Ищи агентов с маркером `source: claude-code-practices` в frontmatter
- Покажи список найденных и спроси для КАЖДОГО:
  > Агент «[имя]» создан пакетом. Удалить? Учтите: если вы используете этого агента в других проектах — он исчезнет и там. Это действие необратимо.
- Агентов БЕЗ маркера — не предлагать удалять вообще

**3. Навыки — аналогично агентам.**
- Удалять только навыки, созданные пакетом (по маркеру или известным именам: silent-mode)
- Спрашивать подтверждение для каждого

**4. Память (MEMORY.md) — НИКОГДА не предлагать удалять.**
- Пользователь мог записать туда ценные сведения, не связанные с пакетом
- Максимум — предложить: «Хотите, я уберу из MEMORY.md записи, сделанные при настройке пакета? Ваши собственные записи останутся.»

**5. Блок статуса настройки — удалить.**
- Блок `<!-- [claude-code-practices] BEGIN: setup-status -->` можно удалить безопасно
